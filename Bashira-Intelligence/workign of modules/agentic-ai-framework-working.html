Agentic Architecture

How the Complete Agentic AI Framework Works in Bashira Intelligence

This document explains the full agentic AI system in Bashira Intelligence as it actually operates across the backend: API entrypoint, orchestration, LangGraph state machine, memory, routing, retrieval, SQL generation, self-healing, predictive branching, public-answer composition, and explainability. The focus is on how the complete multi-agent framework behaves as one governed system.

fastapi entry langgraph state machine dspy agent modules self-healing and memory
Core point. Bashira is not a single chatbot calling one prompt. It is an agentic runtime where different specialized agents operate over a shared state, are routed conditionally, can skip branches, can repair failures, and can switch between deterministic SQL intelligence and predictive decision intelligence.

LangGraph

The workflow is implemented as a stateful graph, not a linear prompt chain.

Agent Modules

1. What Makes This System Agentic

A system becomes agentic when it stops being one static inference call and instead behaves as a governed decision process. Bashira qualifies as agentic because it has:

  • specialized agents with distinct responsibilities
  • a shared state object passed through graph nodes
  • conditional routing based on intermediate outputs
  • conversation-aware context resolution
  • retrieval before generation
  • validation and self-healing loops
  • memory/checkpoint support across sessions
  • multiple execution paths depending on question type

In other words, the framework does not just “answer questions.” It reasons about what kind of question was asked, what context is missing, what data is relevant, whether SQL is safe, whether results require a chart or a follow-up, and whether the problem should switch from deterministic SQL to predictive analysis.

2. System Entry: FastAPI as the Front Door

The backend entry layer is a FastAPI server. It exposes the main API routes and lazily initializes heavy services so the system does not pay every startup cost immediately.

LayerRole
/api/queryMain natural-language entrypoint into the orchestrator.
/api/drill-downExecutes drill-down actions from charts.
/api/knowledge-graphExposes graph/ontology structure for the graph UI.
warm-up routinesPreload Causal Command, Forecast, Data Integrity, and Command Center views.

The API server is not the agent itself. It is the infrastructure boundary that receives the question, ensures services exist, and delegates execution to the orchestrator. 

Frontend -> FastAPI /api/query -> PipelineOrchestrator -> LangGraph Agent -> response

3. Orchestrator: The Operational Brain

The orchestrator is the application-level brain that assembles the runtime. On initialization it configures DSPy, connects to Neo4j and SQL Server, builds the BM25 index, initializes the graph agent, and wires in the checkpointing layer.

It also acts as the stable API-facing contract. The frontend does not need to know which node ran, which branch was taken, or whether the checkpointer failed and fell back. The orchestrator handles that operational complexity.

Responsibilities

  • configure DSPy with the active LLM provider
  • connect database clients
  • build the BM25 retrieval index from the columns CSV
  • create the BashiraGraphAgent
  • attach a checkpointer
  • invoke the graph with the current session and chat history
  • normalize the final state into a PipelineResult for the API

Why This Layer Matters

Without the orchestrator, the system would be a collection of agents with no stable operating contract. The orchestrator is what turns those parts into a product-grade runtime: controlled initialization, controlled invocation, controlled output, and controlled fallback when infrastructure such as the remote checkpointer fails.

4. Checkpointing and Session Memory

Bashira supports checkpointed execution. Preferred mode is a Postgres-backed checkpointer. If that is unavailable, the system falls back to an in-memory saver.

This is important because the graph keeps conversational context and reasoning state across turns. The checkpointer stores graph state by session or thread ID, allowing follow-up questions and smarter drill-down behavior.

Operational advantage. If the remote Postgres checkpointer drops mid-request, the orchestrator can retry the same graph invocation using an in-memory checkpointer, so the user query still completes instead of failing hard.

5. Shared Graph State: The Core of Agent Coordination

The graph does not pass text from one prompt to another blindly. It passes a structured state object. This is one of the main reasons the framework is agentic rather than just prompt-chaining. 

AgentState includes:
- messages
- intent_data
- schema_context
- retrieval_metadata
- session_context
- current_sql
- sql_is_safe
- sql_confidence
- sql_reasoning
- columns, rows, total_rows
- execution_error
- reasoning_steps
- chart_config
- requires_followup
- followup_prompt
- response_type
- clarification_prompt
- response_mode
- predictive_context
- answer_text
- risk_label
- predictive_summary

Every node reads part of this state and writes part of this state. That means no node needs to “remember” everything alone. The intelligence emerges from the coordinated mutation of shared state.

6. Node Graph: The Actual Agent Workflow

The workflow is explicitly built as a LangGraph StateGraph. The important point is that it is not fixed to one path. Conditional routing determines where the state goes next. 

Entry -> resolve_context
      -> extract_intent
      -> conditional route:
           sql_only   -> retrieve_schema -> generate_sql -> execute_sql -> analyze_data -> {followup | chart}
           predictive -> predictive_scan -> compose_public_answer -> end

There is also a clarification branch after SQL generation if the system decides confidence is insufficient or required schema is missing.

NodePurpose
resolve_contextRewrite vague follow-ups into self-contained questions using conversation context.
extract_intentExtract entities, expand abbreviations, classify query type, and decide response mode.
retrieve_schemaRun hybrid retrieval over ontology/schema assets.
generate_sqlProduce SQL using DSPy + few-shot patterns.
clarification_nodeAsk for a clearer question when the graph cannot safely proceed.
execute_sqlValidate and execute SQL, including self-healing if needed.
analyze_dataDecide whether results need a chart or conversational follow-up.
conversational_followupTransform list-like outputs into a more explainable intermediate response.
generate_chartBuild chart configuration and drill-down actions.
predictive_scanCollect CPU predictive intelligence when the question is predictive.
compose_public_answerGenerate an executive-facing answer card from predictive context.

7. Context Resolution: Why Follow-Ups Work

One of the most important agentic behaviors in Bashira is context resolution. Many users do not ask a perfectly formed new question. They say things like “show me that”, “yes give me details”, “what about it”, or “same for this week”.

Bashira handles this with a dedicated DSPy context-resolution signature plus rule patterns. The system looks at recent chat history, identifies whether the latest user message is vague, and if so rewrites it into a fully self-contained question. 

conversation history + vague follow-up -> resolved self-contained question

This is a major reason the system feels conversational instead of brittle. The user does not have to restate the entire context every turn.

8. Intent Agent

The IntentAgent is a lightweight DSPy module that extracts entities, expands abbreviations, adds column hints, and classifies the broad shape of the query before retrieval starts. 

Input:
- user question
- available tables context

Output:
- entities
- query_type
- expanded_question
- column_hints

This step improves downstream precision because retrieval does not begin from raw natural language alone. It begins from a partially structured interpretation.

9. The First Major Routing Decision: SQL or Predictive

Bashira does not push every question through the same path. It uses a classifier that decides whether the question should stay on the deterministic SQL path or switch to the predictive path.

The classifier is intentionally conservative. It keeps deterministic KPI and SQL-answerable questions on the SQL branch and reserves the predictive branch for true forecasting, root-cause, intervention, probability, or counterfactual questions.

BranchTypical Question Type
sql_onlycurrent metrics, counts, lists, rankings, deterministic aggregations, exact current status
predictiveprobability of delay, root causes, interventions, scenarios, simulations, management actions

This routing is a critical agentic capability. It prevents expensive or inappropriate predictive tooling from being invoked for questions that SQL can already answer cleanly.

10. Retrieval Agent

The retrieval step is hybrid and schema-aware. It combines three retrieval channels:

  • Neo4j vector search over semantic embeddings
  • Neo4j fulltext search over ontology/schema terms
  • BM25 search over schema and column descriptions

Then the results are merged using Reciprocal Rank Fusion. The retrieval output is not just a bag of matches. It becomes:

  • a structured set of relevant columns
  • table context
  • join path hints from the knowledge graph
  • a schema context string for the SQL agent
question -> vector/fulltext/BM25 -> RRF -> schema context + join paths

Retrieval before generation is one of the key reasons the framework is reliable. The model is not asked to invent SQL from memory. It is grounded first.

11. SQL Agent

The SQL agent is a DSPy-based generation module with manual few-shot examples and anti-hallucination prompt rules. It is not freeform code generation. It is constrained SQL synthesis against the schema context supplied by retrieval.

Important properties of this agent:

  • uses DSPy ChainOfThought style reasoning
  • sees retrieved schema and join hints
  • has domain-specific few-shot examples
  • can emit an insufficiency signal instead of inventing columns

This is what converts the product from semantic search into executable analytics.

12. Validator Agent and Self-Healing

The validator is one of the most important agentic components because it closes the loop between generation and execution. It validates SQL against four categories:

  1. security rules
  2. schema correctness
  3. BM25 anti-hallucination checks
  4. runtime execution success

If the SQL fails, the system does not simply stop. It can re-prompt the SQL agent with error context and attempt an automatic correction. That is genuine agentic repair behavior. 

generated SQL -> validate -> fail -> feed error context back -> regenerate -> retry
Why this matters. Many NL2SQL systems break because they generate once and hope for the best. Bashira uses a governed self-healing loop, which is much closer to an actual agent than a one-shot text model.

13. Analyze Data, Follow-Up Logic, and Chart Agent

After SQL execution, the framework does not blindly dump rows. It inspects the result shape and decides what kind of response is appropriate. If the result needs visualization, it routes into chart generation. If the result is more list-like or conversational, it can enter a follow-up node first.

The ChartAgent then turns result columns, row counts, and the original question into a chart configuration, including possible drill-down actions. This means the output layer is also agentic: it adapts rendering to the structure of the data.

14. Predictive Scan Agent

The PredictiveScanAgent only runs when the response mode is predictive. It is intentionally skipped for ordinary SQL questions.

This agent pulls from the CPU predictive stack, including:

  • ForecastEngine for portfolio and well forecast context
  • CPU ML microservice on port 8050 for delay probabilities, SHAP insights, and causal hints

It assembles a predictive context object containing forecast summaries, risk summaries, causal insights, intervention ideas, and structured scan lines. So the predictive branch is not a generic LLM answer. It is a retrieval-and-assembly layer over real predictive services.

15. Public Response Agent

Once predictive context exists, the PublicResponseAgent composes an executive-facing answer. This layer exists because raw model context is too technical for the frontend answer card.

It takes:

  • the original question
  • SQL summary if present
  • forecast summary
  • causal summary
  • intervention summary

And returns:

  • answer_text
  • risk_label
  • predictive_summary lines for the UI

Importantly, it is explicitly designed not to expose internal chain-of-thought, SQL, or debugging artifacts. This is the public-facing narrative layer.

16. Explainability Through Reasoning Steps

Bashira tracks reasoning steps as structured metadata. Each step can record:

  • step name
  • status
  • detail
  • duration
  • extra metadata

This gives the system a process trace without exposing hidden chain-of-thought. In practical product terms, it means the UI and logs can explain:

  • which branch ran
  • whether retrieval succeeded
  • whether SQL needed healing
  • whether predictive scan was available
  • how the final output mode was chosen

17. Full End-to-End Agentic Flow

1. Frontend sends question to /api/query
2. API server lazy-loads orchestrator if needed
3. Orchestrator gathers session context and chat history
4. LangGraph invocation starts with resolve_context
5. Vague follow-ups are rewritten into self-contained questions
6. Intent agent extracts entities and query shape
7. Classifier chooses sql_only or predictive mode
8. If sql_only:
     a. Retrieval agent builds schema context
     b. SQL agent generates SQL
     c. Validator checks security/schema/runtime
     d. Self-healing runs if necessary
     e. SQL executes
     f. Data analysis chooses chart or conversational follow-up
     g. Chart agent builds final visualization config
9. If predictive:
     a. PredictiveScanAgent gathers forecast/risk/causal context
     b. PublicResponseAgent builds executive answer
10. Orchestrator normalizes final state into API response
11. Frontend renders chart, table, predictive card, or clarification

19. Bottom Line

The complete Bashira agentic AI framework works because it separates responsibilities and connects them through a governed graph runtime. The system understands conversational context, classifies the question, retrieves the right schema, generates and repairs SQL, executes safely, adapts the output format, and can switch into a predictive decision branch when required.

That is what makes it a real agentic architecture: specialized agents, shared state, memory, routing, validation, recovery, and mode switching, all operating together as one controlled intelligence system.
