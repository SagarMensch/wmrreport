1) What Decision Studio Is Architecturally

Decision Studio is not the same surface as Predictive Studio. Predictive Studio is a live well forecasting workspace built around ForecastEngine. Decision Studio is an institutional model package viewer. It loads a published set of trained-model artifacts, turns them into executive and analyst-facing charts, and lets the user inspect the evidence behind portfolio risk, predictive drivers, completion probability, and rig performance. Only when the user opens the embedded SequelForecastStudio does the tab move into live backend inference mode.

Layer Main File Purpose
Tab shell web/src/components/DecisionStudio.tsx Loads artifact files, computes KPI counts client-side, renders 2D and 3D analytics board
Tooltip explainer web/src/app/api/explain/route.ts Builds context-aware explanations from current artifact snapshot, with optional Groq augmentation
Live forecast overlay web/src/components/SequelForecastStudio.tsx Calls /api/predict?action=... for real-time prediction, nightly refresh, anomalies, and portfolio summary
Backend predictive engine predict_service.py + feature_engine.py Runs trained ML inference, survival prediction, portfolio summary, and anomaly tracking

2) End-to-End Runtime Flow

The main Decision Studio path is artifact-first. It loads model outputs from static files, derives portfolio views in the browser, and renders them immediately. The only always-live service call in the top board is the explanation route for hover help. The full online predictive path begins only when the user opens the live forecast overlay.

3) Static Artifact Package Loaded by the Tab

  • /wmr_results/ag_metrics.json for RMSE, MAE, R2, MAPE, AUC, Brier, and C-index summary.
  • /wmr_results/risk_scores.csv for portfolio risk tier counts and rig aggregation.
  • /wmr_results/feature_importance.csv for top predictive drivers.
  • /wmr_results/priority_wells_final.csv for the main well table and predicted completion dates.
  • /wmr_results/training_manifest.json for thresholds, ensemble members, and survival metadata.

This means the tab can render fast without forcing a live model round-trip for every chart.

4) Why the Artifact Strategy Exists

  • Speed: heavy computation is already baked into CSV, JSON, and PNG outputs.
  • Consistency: every viewer sees the same validated package, thresholds, and evidence.
  • Demo resilience: the core board still works even if backend services or VPN-only sources are unavailable.
  • Governance: published metrics and plots are tied to a training manifest rather than hidden runtime state.

6) Visualization Stack: 2D and 3D

Decision Studio maintains two visualization modes driven by the same underlying arrays:

  • Recharts 2D mode: PieChart, BarChart, RadarChart, AreaChart, and standard Cartesian views.
  • Plotly 3D mode: mesh3d prisms, scatter3d rig positioning, and a portfolio density surface.

The code constructs custom 3D bars using Plotly mesh3d rather than using simple flat columns. This is why the 3D mode can encode depth and produce a more volumetric presentation of risk, rig performance, and feature importance.

8) Training Manifest: What It Reveals About the Model Package

training_manifest.json is the governance spine of Decision Studio. It records what was produced in the training run, the thresholds for risk tiers, survival covariates, and the ensemble members behind the published model package.

Manifest Field Observed Meaning Why It Matters
best_model DecisionStudio_Ensemble_CatBoost_XGBoost Shows the primary package is an ensemble, not a single regressor or classifier
tier_thresholds Watch, High Risk, Critical cutoffs Explains how wells are bucketed into the visible tier counts
risk_metrics AUC and Brier score Shows rank quality and calibration quality for the risk model
survival_metrics C-index and Cox features used Shows that completion timing is modeled as a survival problem, not just a regression problem
progress_ensemble_members, weeks_ensemble_members, risk_ensemble_members CatBoost and XGBoost families with weights Makes the model package inspectable at the ensemble composition level

12) /api/explain: Interactive Explanation Layer

The small info controls inside Decision Studio are not hardcoded tooltip strings. They call /api/explain, which loads the same artifact package, builds a current snapshot, and then returns grounded explanations for terms like Portfolio, Risk Distribution, Rig Profile, Predictive Drivers, RMSE, AUC, or Completion Drivers.

Step What Happens
1 Client posts a term and optional context to /api/explain
2 Route loads current artifact snapshot from public/wmr_results and caches it for 60 seconds
3 Route computes facts: well count, tier counts, thresholds, top drivers, survival covariates, validation metrics
4 Route either builds a deterministic fallback explanation or calls Groq if API keys are present
5 Returned explanation is cached by term and context, so repeated hovers are fast

13) SequelForecastStudio: The Live Predictive Overlay

Inside Decision Studio there is a separate embedded product called SequelForecastStudio. This is the live action layer. When opened, it switches from static artifact review to backend inference and operations:

Frontend Action Proxy Route Backend Route Effect
Target well forecast /api/predict?action=single POST /predict/single Runs real-time single well forecast using SQL or offline artifacts
Nightly evaluation /api/predict?action=refresh POST /predict/refresh Batch evaluates active wells and updates anomaly state
Full recalibration /api/predict?action=full POST /predict/full Queues full retrain workflow instructions
Anomaly feed /api/predict?action=anomalies GET /predict/anomalies Returns tier changes detected in prior refresh cycles
Portfolio summary /api/predict?action=portfolio GET /predict/portfolio Returns portfolio-level predictive summary from the ML service
Model info /api/predict?action=model-info GET /predict/model-info Exposes model readiness and artifact health

14) What Predictive Tech the Live Overlay Uses

  • FeatureEngine: engineers 58 features from well records and history
  • AutoGluon: tabular progress prediction stack
  • Random Survival Forest: estimates time-to-completion
  • Exact risk formula: combines progress, velocity, schedule, and gap components
  • AnomalyTracker: tracks changes in risk tier between refresh cycles

So Decision Studio is not disconnected from the product's real ML. It simply presents the published layer first, then lets the user cross into live inference when needed.

15) Why the Two-Layer Design Is Smart

  • Board speed: top-level analytics open fast from published artifacts.
  • Evidence quality: users see validated SHAP, survival, and diagnostic outputs.
  • Operational escalation: when they want action, they can open the live forecast engine.
  • Demo resilience: the board still works even if some live services are slow.
  • Institutional feel: this is closer to a real investment or operations review pack than a noisy transactional dashboard.

16) Decision Lab Wrapper Around Decision Studio

There is also a higher-level wrapper called DecisionLab.tsx. It provides three interchangeable views:

  • Decision Studio - the institutional analytics board described in this document
  • Predictive Forecast - the live Predictive Studio surface
  • Causal Why - the Causal Command surface

This means the product is organized around three analytical modes: descriptive model package review, predictive forecast operations, and causal intervention reasoning.

18) Most Important Architectural Insight

Decision Studio is a published intelligence package, not just another analytics tab. Its job is to turn model outputs into an institutional review surface: tier thresholds, validation quality, rig comparison, driver ranking, SHAP evidence, survival evidence, and portfolio prioritization.

The tab becomes especially powerful because it does not stop at passive viewing. Once the user needs deeper operational action, it can open the live SequelForecast workspace and hand off into the backend predictive engine. That two-step design gives both presentation quality and operational depth.
