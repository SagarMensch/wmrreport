1) What Causal Command Is Architecturally

Causal Command is not a generic analytics tab. It is a decision-operations layer built to turn raw portfolio status into intervention-ready recommendations. The service assembles a cross-system feature matrix from multiple SQL sources, computes a live fast decision layer, and then optionally enriches the result with Bayesian posterior counterfactual summaries. The user-facing tab presents all of this through three modes: Command, Trace, and Engine.

Layer Main File Purpose
Tab UI web/src/components/CausalCommand.tsx Fetches workspace, renders command bar, plots, evidence trace, model state, and selected-well recommendations
Frontend proxy web/src/app/api/causal/route.ts Forwards to backend /api/causal/command with a long timeout for deep analysis

Posterior engine causal_stan_service.py Runs CmdStan MCMC when toolchain exists, otherwise falls back to Laplace approximation with diagnostics

3) Frontend Surface and View Modes

The user-facing component keeps three distinct modes:

  • Command: selected well workspace, scenario frontier, portfolio pressure map, top drivers, and management summary.
  • Trace: source trace rows that show where facts and model signals came from.
  • Engine: model basis, diagnostics state, coverage, and analysis-mode metadata.

The command bar is not just cosmetic. It interprets short commands like highest delay well, best action, act now, weak support, and open support to switch the currently selected object or mode.

4) Frontend Connectivity Behavior

The component uses getCausalEndpoint() to decide how to reach the service:

  • On localhost it can call http://127.0.0.1:8005/api/causal/command directly.
  • In deployment it calls the web proxy route /api/causal.

The proxy itself uses a long timeout of 180 seconds because deep posterior refresh and large workspace builds can take materially longer than a normal CRUD call.

7) Target Variable Used by the Decision Engine

The core supervised target is target_delay_days. It is not an arbitrary label. It is computed from expected versus actual rig-off logic:

  • If expected rig-off and actual rig-off both exist, delay is max(actual - expected, 0).
  • If actual rig-off is still missing but expected rig-off is already in the past, delay becomes today - expected.
  • Otherwise delay is zero.

That makes the entire tab operationally grounded. Recommendations are not about abstract risk only; they are tied to concrete schedule slippage pressure.

8) Fast CPU Analysis Layer

_run_fast_cpu_analysis() is the first usable layer of the product. It attempts to train and score a LightGBM regressor over the current joined feature matrix.

  • If there is enough data and LightGBM is available, the service fits a supervised model on live joined features against delay days.
  • If SHAP can also be computed, the service uses SHAP values to produce top drivers and per-well root-cause stacks.
  • If the data is too small or dependencies are unavailable, it drops into a rule-based CPU analysis instead of failing the tab.

This is why the tab can still produce a governed recommendation deck even under constrained local conditions.

9) Root Causes and Scenario Generation

Once the CPU layer has a baseline delay prediction for each well, the service builds action scenarios. This is not a generic "what-if" toy. It is a governed scenario generator built on peer comparisons and model re-scoring.

Step What the Service Does
1 Selects peers for the well using cluster and well-type, widening to portfolio peers if support is too thin
2 Chooses actionable features such as overdue backlog, daily completion rate, five-week plan pressure, rig move days, weekly velocity, or linked productivity
3 Finds a better peer-median target value for that feature
4 Applies that changed value to a copy of the well feature row
5 Re-scores the modified row through the model and measures the resulting delta in delay days
6 Builds a governed scenario object with label, description, support cases, baseline delay, new delay, and assumption note

10) Why Peer Medians Matter

The intervention targets are not random slider values. They come from better-performing comparable peers, which makes the resulting action labels more believable operationally.

  • Cluster and well-type peers are preferred.
  • If support is too small, the scope widens gradually.
  • The selected target becomes the median of better-performing peers.

11) Confidence and Action Status

Confidence is not a pure posterior interval. The current product uses a first-pass hybrid heuristic:

  • Support cases contribute to confidence.
  • Signal quality depends on how many strong root causes were found.
  • Delta size matters; positive or zero delta yields weak support.

Action status is then derived:

  • Act Now if recovery is material and support is adequate
  • Candidate for weaker but still useful scenarios
  • Observe when evidence is too thin

12) Workspace Objects Built for the UI

  • portfolio_brief: management message, attention-now counts, KPI cards, top opportunity
  • wells: per-well baseline delay, best action, scenarios, root causes, support, confidence, decision score
  • intervention_ladder: ranked opportunities for action
  • analysis_basis: model name, mode, rows, features, R2, RMSE, posterior status

13) Bayesian Layer: StanCounterfactualService

The deeper analysis layer is handled by StanCounterfactualService. It is designed as an institutional-grade posterior service:

  • Preferred path: CmdStan MCMC with 4 chains, warmup, and posterior sampling.
  • Fallback path: Laplace approximation when CmdStan or build tools are unavailable.
  • Diagnostics: MCMC quality stats, toolchain readiness, and LOO-CV reporting where available.
  • Runtime label: surfaced back into the workspace as part of data health.

The important product idea is that the Bayesian layer does not block the deck. The fast CPU workspace is returned first, and the Bayesian worker updates the cached workspace in the background.

14) CmdStan Versus Laplace Approximation

Mode Behavior When Used
CmdStan MCMC Compiles the Stan model, samples a full posterior, records diagnostics like R-hat and ESS, and can run LOO-CV When CmdStan and Windows toolchain dependencies are available
Laplace approximation Computes a MAP-centered uncertainty approximation and still returns posterior-style summaries When MCMC compilation or toolchain visibility is not available locally

This is why your logs showed the product continuing to work even when Stan compilation failed: the service was designed to preserve the causal deck through Laplace approximation.

15) Background Refresh and Cache Strategy

  • The service caches the assembled workspace for 600 seconds.
  • The returned workspace always includes cache age and whether Bayesian refresh is currently in progress.
  • If a fresh Bayesian result already exists and is young enough, it is reused.
  • If not, the service starts a background thread and returns the CPU deck immediately.

This design balances responsiveness and sophistication. Executives do not wait for deep sampling just to get a ranked action deck.

16) Audit and Data Health Layer

Causal Command does not hide data gaps. It produces explicit workspace metadata:

  • coverage_cards: join coverage for job progress, plan snapshot, daily task, SAP drilling, ActivityTaskPlan, and PH productivity links
  • data_health: live wells, historical wells, mean progress, delayed wells, Bayesian runtime, analysis mode, CPU model status
  • audit_questions: a library of answered or gap-marked questions about portfolio state, backlog, stalled wells, rig performance, and productivity
  • gaps: explicit notes on grain limitations, such as project-scoped ActivityTaskPlan linkage or supervisor-scoped PH productivity linkage

This makes the tab more credible in front of a client because it shows what the model knows, what it joined, and what limitations still remain.

19) Most Important Architectural Insight

Causal Command is a governed intervention engine. It does not try to answer everything through one monolithic model. Instead it separates the problem into: live SQL feature assembly, a fast decision-grade CPU deck, and a slower posterior counterfactual layer that enriches the same workspace when ready.

That split is exactly why the tab is practical. It can serve executives immediately, remain operational on a local offline clone, and still offer a deeper Bayesian story when the heavier statistical layer finishes refreshing.
