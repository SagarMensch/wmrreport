7) BM25 in This System (Detailed)

BM25 is a lexical retrieval model used here to score how strongly schema documents match query words. In Bashira, each schema document is generally a column-level record (table, column, description, data type). This gives strong exact-match behavior for domain terms such as rig identifiers, milestone names, and business abbreviations.

BM25 Component How It Works Why It Helps NL2SQL
Term frequency (TF) Higher score when query terms appear in a schema doc multiple times Boosts strongly relevant columns for repeated domain terms
Inverse document frequency (IDF) Rare terms get higher weight than common terms Prioritizes uncommon operational terms over generic words
Length normalization Prevents long descriptions from winning only by size Keeps concise, precise schema docs competitive

Operationally: BM25 catches exact wording and nomenclature better than pure embeddings when users ask with enterprise-specific terms.

8) What MiniLM Means Here

The embedding model is all-MiniLM-L6-v2 (384-dimensional vectors). It converts both the user query and schema text into vectors, then Neo4j vector search returns semantically close columns/tables even if the wording is different.

  • L6: compact transformer depth for low latency.
  • 384 dimensions: smaller vector size than larger models, good speed-memory tradeoff.
  • Sentence-level semantics: catches intent similarity, not only exact token overlap.
  • Production relevance: useful for synonyms, paraphrases, and noisy manager language.

MiniLM does not replace BM25. It complements it. Semantic recall comes from embeddings; lexical precision comes from BM25/fulltext.

9) Why Hybrid Retrieval Is Necessary

Pure vector retrieval misses exact IDs and hard tokens. Pure lexical retrieval misses paraphrase intent. Bashira combines three independent retrieval channels, then fuses them using RRF (Reciprocal Rank Fusion).

Channel Strength Weakness
Neo4j Vector Semantic similarity Can miss exact enterprise tokens
Neo4j Fulltext Lucene keyword matching Sensitive to wording form
BM25 Strong lexical ranking Limited semantic generalization

10) NLP to SQL: Why It Is Difficult in Real Systems

NL2SQL is hard because user language is ambiguous, schema is imperfect, and business meaning often depends on hidden assumptions. In industrial operations datasets, this is amplified by mixed naming styles, sparse fields, legacy aliases, and proxy-only metrics.

Difficulty Example Pattern Bashira Control
Ambiguous intent "show risk this week" without scope or metric definition Intent classification + clarification prompts + context resolution
Schema mismatch User asks for a fact not present in local clone INSUFFICIENT_SCHEMA path and explicit proxy questions
Hallucinated columns LLM invents non-existent fields INFORMATION_SCHEMA validation + BM25 schema check + self-heal retries
Unsafe SQL risk Destructive or malformed query generation Forbidden keyword/pattern checks + read-oriented enforcement
Follow-up ambiguity "yes show details" after previous answer Context rewrite node before intent routing

11) How Bashira Converts Natural Language Into Reliable Analytics

  1. Normalize language: abbreviations are expanded, query intent is classified, and question scope is stabilized.
  2. Retrieve grounded schema: vector + fulltext + BM25 candidates are fused with RRF and enriched with join paths.
  3. Generate constrained SQL: SQL agent writes query only from retrieved schema context, with domain few-shot guidance.
  4. Validate before trust: security validation, schema validation, and runtime execution checks are enforced.
  5. Self-heal on failure: SQL is regenerated with explicit error context up to configured retry limits.
  6. Render decision output: chart engine selects best format and creates drill-down actions.
  7. Predictive augmentation (if required): Forecast + CPU ML stack adds risk, drivers, and intervention summary.

14) Algorithm-Level Mechanics (BM25, MiniLM, RRF, Risk Scoring)

14.1 BM25 lexical relevance

BM25 ranks schema documents by lexical importance. For query term q_i and schema document D, the score used in principle is: score(D,Q)=sum(IDF(q_i) * (f(q_i,D)*(k1+1)) / (f(q_i,D)+k1*(1-b+b*|D|/avgdl))). This helps Operations when users mention exact enterprise tokens (rig IDs, task codes, milestone names).

14.2 MiniLM semantic retrieval

all-MiniLM-L6-v2 embeds query and schema text into 384-dim vectors. Neo4j vector index compares embeddings with cosine similarity: cos(u,v)= (u.v) / (||u|| ||v||). This recovers relevant columns even when user wording is paraphrased.

14.3 Reciprocal Rank Fusion (RRF)

The system merges ranked outputs from vector, fulltext, and BM25 using: RRF(d)=sum(1/(k + rank_j(d))). This avoids dependence on one retrieval channel and improves robustness across wording styles.

14.4 Predictive pressure score composition

In predictive mode, a composite risk/pressure signal is formed from multiple drivers: schedule slip, plan gap, stage stall, queue contagion, anomaly signal, and confidence gap. A practical expression is: Risk = w1*slip + w2*plan_gap + w3*stage_stall + w4*queue + w5*anomaly + w6*(1-confidence), where weights are calibrated from domain behavior and bounded for stable ranking.

14.5 Confidence scoring logic

Confidence is lowered when schema coverage is weak, retries are high, or data completeness is poor. It is increased when retrieval sources agree, SQL validates on first pass, and returned rows have healthy non-null coverage. This confidence is then fed back into response language and intervention priority.

15) NL2SQL Generation and Self-Heal (Detailed Execution)

  1. Question normalization: abbreviation expansion and context completion create a stable query form.
  2. Schema grounding: hybrid retrieval builds a candidate set of tables, columns, and join hints.
  3. Constrained generation: SQL agent produces query from grounded context and client few-shot templates.
  4. Static validation: reject unsafe verbs and non-read patterns before SQL ever runs.
  5. Schema validation: table/column checks against local SQL metadata and retrieval context.
  6. Runtime execution: execute on SQL Server clone; capture runtime failures.
  7. Error-aware regeneration: feed failure signature back to SQL agent and retry under bounded attempts.
  8. Result shaping: detect if chartable; if not chartable, provide numeric/text answer with follow-up prompts.

This loop is why the system can recover from invalid object names, missing columns, and malformed grouping clauses while staying explainable.
