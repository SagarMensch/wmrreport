How the Rig Operations Tab Works

Rig Operations is the fleet-pressure control tower of the product. It does not just show rigs and wells. It computes a well-level operational risk score, injects queue contagion and CPU outlier detection, rolls those signals upward into rig-level pressure scores, and then exposes that stack through a multi-surface frontend built for drill-down, chart focus, and quantitative explanation.

1) Runtime Path

The React surface is RigOperations.tsx. It loads its payload through useCommandCenterData("rig_operations"), which calls the shared Next.js proxy route /api/command-center?action=rig_operations. That proxy forwards to the backend route /api/command-center/rig-operations, which returns CommandCenterService.get_rig_operations_view().

The architectural consequence is that Rig Operations is part of the same command-center family as Portfolio and Smart Alerts. It shares the same operating frame but applies a different scoring and aggregation philosophy.

2) Why This Tab Exists

A portfolio view tells leadership which projects are under pressure. A rig view answers a different question: which rig queue is about to propagate delay through multiple wells, and how recoverable is it?

  • Portfolio aggregates by project
  • Rig Operations aggregates by rig queue
  • The rig becomes the operative contagion channel

4) Well-Level Feature Preparation

For every well in the operating frame, the service constructs a prepared record. This is not the final score yet. It is a feature object containing schedule, trend, stage, governance, and queue information.

  • delay_days and rig_on_delay_days
  • current_month_gap_pct and avg_gap_pct
  • velocity_pct
  • stage metrics and execution-stage subset
  • queue_exposure
  • missing_fields and moc_pending
  • existing well risk prior from the upstream forecast layer

The service also determines the dominant bottleneck by taking the slowest stage within the active execution subset.

5) Stage Logic and Bottleneck Selection

The stage system is intentionally phase-aware. The service first computes completion for: Location Prep, Engineering, Construction, Commissioning, Flowline, and OHL.

Then _select_execution_stage_metrics() narrows attention depending on overall progress:

  • Very early wells focus on Location Prep and Engineering
  • Mid-stage wells focus on Engineering and Construction
  • Later wells focus on Construction and Flowline
  • Near-end wells focus on Flowline and Commissioning

This prevents irrelevant late-stage metrics from dominating an early-stage well, and vice versa.

6) Base Score Formula

Rig Operations first computes a base_score for each well. This is a weighted operational-pressure score before queue contagion and anomaly detection are injected.

base_score = 0.30 * schedule_component + 0.22 * trend_component + 0.20 * stage_component + 0.18 * prior_component + 0.10 * data_component

Component How It Is Computed What It Means
schedule_component Rig-off delay, arrival delay, and a penalty if the rig is already delayed or overdue Direct schedule pain
trend_component Current month gap, average weekly gap, and negative velocity Execution deterioration against plan
stage_component Average stage gap, stage imbalance, and penalty for immature bottleneck stage Workflow stall intensity
prior_component Upstream forecast risk score, clipped to 0-100 Known portfolio-level prior risk
data_component Missing required fields and pending MOC penalties Operational confidence and governance drag

This is the deterministic analytical core of the tab. It answers: how much pressure would this well generate even before considering fleet contagion and outlier behavior?

7) Queue Exposure and Rig Base Map

The model then adds a rig-level contagion channel. First it builds an active_queue_map that lists active wells under each rig. For each well:

queue_exposure = max(0, queue_size - 1) for active wells.

Then it computes a rig-level background pressure called rig_base_map using the top three well base scores, the rig-wide mean base score, and the rig’s maximum schedule component. That means a well is not judged only on its own row. It is judged partly in the context of the rig carrying it.

8) CPU Isolation Forest Layer

Rig Operations contains a real CPU machine-learning layer through scikit-learn’s IsolationForest. It is used only when there are enough feature rows and enough variation to make anomaly detection meaningful.

  • n_estimators = 240
  • random_state = 42
  • contamination adapts to sample count and is clipped between roughly 0.08 and 0.18

Features include progress, delays, plan gap, negative velocity, stage gap, stage imbalance, risk score, queue exposure, and missing fields. Raw scores are normalized to 0-100. Wells are flagged when the model predicts outlier behavior or when the normalized anomaly score is high enough.

9) Final Operational Risk Score

After the deterministic base score, queue contagion, and anomaly layer are available, the service computes the final ops_risk_score.

ops_risk_score = min(99, 0.78 * base_score + 0.14 * queue_component + 0.08 * anomaly_score)

Completed wells are treated differently and receive only a low residual score from their prior component. Active wells, however, are evaluated in full. This is why the tab can distinguish between a risky isolated well and a risky well embedded in a stressed rig queue.

Term Role
base_score Intrinsic schedule and execution pressure of the well
queue_component How much extra pressure comes from sharing a stressed rig queue
anomaly_score How unusual the well looks relative to fleet execution pattern

10) Recovery Confidence Formula

The tab does not stop at risk. It also computes recovery_confidence_pct, which estimates how recoverable the current situation is.

recovery_confidence_pct = clamp(92 - 0.62 * ops_risk_score + 1.6 * positive_velocity + 0.08 * progress_pct - min(16, 1.4 * queue_exposure), 8, 96)

So recovery confidence improves when:

  • velocity is positive
  • progress is already materially advanced

And it falls when:

  • the operational risk score is high
  • queue exposure is large

This gives the tab a second axis: not just danger, but recoverability.

11) Evidence and Action Generation

Rig Operations reuses the same deterministic action and evidence builders used by other command-center views, but applies them to rig-centered records. That means each well carries:

  • structured evidence such as overdue rig-off, late rig-on, stage stall, queue overlap, anomaly flag, missing fields, and pending MOC
  • action rows such as re-sequencing rig commitment, recovering throughput, unblocking the dominant stage, or reviewing the operational outlier

The actions are not random assistant text. They are selected from the dominant score contributors.

12) Rig-Level Aggregation

Once each well is enriched, the service groups by rig_no and builds rig cards.

Rig Metric Construction Logic
pressure_score Blend of top-three mean well risk and overall mean well risk for active wells on the rig
dominant_bottleneck Stage label whose wells accumulate the most weighted operational pressure
top_actions Deduplicated highest-value interventions harvested from top-risk wells
status Tier derived from pressure score, critical count, delayed count, and anomaly count
wells Detailed roster sorted by risk, delay, arrival delay, and prior risk

The rig is therefore treated as a decision object, not just a grouping key.

13) Returned Payload

The backend returns a compact but information-rich payload:

  • summary: total rigs, active wells, critical rigs, average pressure score, queue-exposed wells, anomaly wells
  • rigs: rig cards with status, pressure, bottleneck, top actions, and full well roster
  • arrival_delay_by_well: top well delay leaderboard
  • bottleneck_mix: distribution of dominant bottlenecks across the fleet
  • engine_label and model_note: model identity and human-readable explanation

This is why the frontend can drive many views without repeated backend calls. One payload contains both executive and detailed quant state.

14) Summary Logic

Fleet-level summary values are themselves aggregates:

  • critical_rigs and high_risk_rigs
  • avg_pressure_score
  • queue_exposed_wells where queue exposure is materially large
  • anomaly_wells flagged by Isolation Forest

15) Frontend Interaction Model

The React component is intentionally built as a control tower rather than a simple document page. Once the payload is loaded, the client performs local slicing with:

  • rig search and severity filters
  • selected rig state
  • selected well state
  • workspace view state
  • dossier view state
  • focus overlays for chart, brief, roster, and wells

The search input uses useDeferredValue so the interface remains responsive while filtering large rig and well sets.

16) Workspace Views

The center workspace is not one chart. It is a switchable quant surface.

  • pressure: top wells ranked by operational pressure
  • drivers: average model-driver decomposition
  • stages: completion versus residual gap across the selected rig
  • queue: rig-on delay and rig-off slip profile
  • playbook: intervention ladder from top actions

This matters because the tab supports multiple operator questions without changing screens.

17) 2D and 3D Chart Strategy

The tab supports both 2D and 3D modes, but they are implemented carefully:

  • 2D views use Plotly for dense analytical reading
  • 3D-style views use a custom SVG renderer called ExtrudedBarsChart

The custom SVG path gives the product a volumetric chart language without requiring a heavyweight scene engine for every rig chart. It also keeps the visuals deterministic and theme-controlled.

18) Model Visibility on the Frontend

The frontend intentionally exposes the quant model instead of hiding it. It shows:

  • engine_label as the active model family
  • CPU Isolation Forest as the outlier layer
  • quant chips for slip, plan gap, stage stall, contagion, outliers, and velocity
  • rig-level and well-level pressure plus recovery metrics

That is why the tab can be used in a demo to communicate both premium UI and actual analytical substance.

19) Why This Tab Creates Real Value

Rig Operations solves a practical operational problem: delays propagate through rigs, not just through projects. If a rig queue is stressed, multiple wells inherit that stress. This tab makes that contagion visible, quantifies it, and prioritizes where intervention is most likely to reduce downstream damage.

  • It separates intrinsic well pain from queue-induced pain
  • It introduces machine-detected execution outliers on CPU, locally
  • It measures both risk and recoverability
  • It elevates rigs into first-class decision objects for intervention planning
  • It exposes evidence and actions directly in the same surface

In effect, this tab is the fleet operating room of the product.

Rig Operations is a hybrid control-tower system: deterministic schedule math, phase-aware stage diagnostics, queue contagion modeling, CPU anomaly detection, and rig-level aggregation, all exposed through a drillable control-tower UI.
