How the Smart Alerts Tab Works

Smart Alerts is the alert-ranking and intervention surface of the product. It does not simply list delayed wells. It constructs a hybrid execution-pressure score for each active well by blending schedule delay, plan-vs-actual divergence, stage stall, project risk prior, anomaly transitions, and governance confidence. The result is a ranked escalation board that is explainable, drillable, and tied back to the local AppMasterDB clone.

1) Runtime Path

The tab is loaded by SmartAlerts.tsx, which uses the shared hook useCommandCenterData("alerts"). The browser fetches /api/command-center?action=alerts, the Next.js proxy maps that to /api/command-center/alerts, and the Python backend returns service.get_smart_alerts_view().

The important architectural point is that Smart Alerts is not fed by a dedicated SQL query written only for this screen. It is composed from a command-center service that fuses portfolio context, integrity context, forecast context, and anomaly context.

2) Freshness Model

The frontend always requests with cache: "no-store", but the backend caches the computed alert payload for roughly 30 seconds. This keeps the board responsive while avoiding unnecessary recomputation of the full alert stack.

  • UI refresh button re-requests the board
  • Backend recalculates after cache expiry
  • Alert ranking therefore behaves near-live without recomputing every paint

4) Job Progress Signal Map

Smart Alerts becomes more than a delay list because it converts weekly operational tracking into a compact per-well signal object. _build_job_progress_signal_map() groups Job Progress by Well ID and computes a five-week mini-time-series.

  • timeline: W1 to W5 plan, actual, and gap values
  • velocity_pct: recent throughput slope, roughly last minus third-last actual value
  • avg_gap_pct: average plan shortfall across the weekly series
  • current_month_plan_pct and current_month_actual_pct
  • current_month_gap_pct: current plan minus current actual, clipped at zero

This is what gives the board temporal intelligence. It can see whether execution is merely behind, or actively deteriorating.

5) Stage Metrics and Bottleneck Detection

The service computes stage completion for six operating stages: Location Prep, Engineering, Construction, Commissioning, Flowline, and OHL.

For each stage it records:

  • value_pct: current completion
  • gap_pct: remaining distance to 100 percent

The slowest stage is selected as the bottleneck. That bottleneck drives owner assignment, action generation, evidence text, and part of the model score. This makes the alert operationally explainable rather than purely statistical.

6) The Hybrid Smart Alert Score

The core of the tab is a five-part additive score breakdown. Each well gets five components, each normalized to a maximum of 10. The final model_score is:

model_score = min(99, 2 * (delay_component + trend_component + stage_component + scope_component + data_component))

Component Formula Logic Meaning
delay_component Rig-off delay, rig-on delay, plus status bump if the rig is already DELAYED or OVERDUE Direct schedule pain
trend_component Current month gap plus negative velocity plus average historical gap Whether the well is falling behind plan over time
stage_component Average stage gap plus a penalty if the slowest stage is very immature Execution bottleneck intensity
scope_component Current well risk score scaled into 0-10 space Project or portfolio prior risk already known upstream
data_component Missing milestone fields, pending MOC, and anomaly count Confidence and governance pressure

This is why the label Hybrid Local Signal Model is technically correct. The board blends deterministic schedule math, short-horizon trend extraction, stage bottleneck logic, upstream risk priors, and governance confidence.

7) Alert Gating and Severity Thresholds

Not every well becomes an alert. If the computed score is weak, the well has no positive delay, and its risk score is still low, the service discards it. That prevents the board from filling with harmless rows.

Severity thresholds are then assigned from model_score:

  • Critical: >= 66
  • High: >= 52
  • Medium: >= 35
  • Low: below that

Finally the board is sorted by severity rank, then by model_score, then by risk_score, and clipped to the top 18 rows. That means the tab is intentionally a ranked intervention board, not a full historical archive.

8) Confidence Score

Each alert includes confidence_pct. This is not model probability in the strict statistical sense. It is an operational confidence heuristic built from evidence density and corroboration.

confidence_pct = min(98, 56 + 5 * evidence_count + 0.25 * model_score + min(10, 4 * anomaly_count))

That means confidence rises when:

  • more independent evidence lines are present
  • the alert score itself is stronger
  • recent risk transitions corroborate that something is changing

The goal is pragmatic: the board should prioritize alerts that are both important and well-supported.

9) Evidence Construction

Evidence is built through _build_alert_evidence(). The service turns raw schedule and governance facts into short, human-readable statements. Evidence can include:

  • rig-off overdue days
  • rig-on lateness
  • current month plan shortfall
  • flat or declining weekly throughput
  • the slowest stage and its progress level
  • rig queue overlap with other active wells
  • recent anomaly transitions
  • missing milestone fields
  • pending MOC approval

These are not generic LLM sentences. They are deterministic statements created from measured conditions.

10) Action Recommendation Logic

Smart Alerts also generates intervention suggestions through _recommend_alert_actions(). The function looks at the dominant contributors in the breakdown and maps them into operational directives.

Dominant Driver Action Style Typical Owner
Delay trend Re-sequence rig commitment Rig Planner
Progress slope Recover weekly throughput Project Controls
Stage stall Unblock the slowest stage Stage lead
Scope risk Escalate project recovery review Project Controls
Data quality Close data confidence gaps Data Steward

If other wells share the same rig, a queue-protection action is also appended. This is why the tab can surface contagion rather than just isolated local pain.

11) Anomaly Tracker Integration

Smart Alerts uses AnomalyTracker as a risk-transition memory layer. This tracker persists tier changes in SQL Server when possible, can fall back to SQLite only if explicitly allowed, and otherwise keeps an in-memory store so the product continues to work.

The tracker records transitions like WATCH -> HIGH_RISK or HIGH_RISK -> CRITICAL, stores the score delta, and exposes recent anomalies back to the command-center service. These anomalies influence:

  • data_component
  • confidence_pct
  • evidence strings
  • system-wide flags and executive insights

This means Smart Alerts is not only snapshot-based. It has memory of change.

12) Frontend Interaction Model

The frontend intentionally downloads one alert payload and then performs local slicing. Search, severity filtering, alert selection, and center-pane tab changes happen client-side. The component uses useDeferredValue for the search input so filtering stays responsive while typing.

The center pane has six operating views:

  • Exposure: board-level severity composition
  • Drivers: per-alert score breakdown
  • Timeline: plan vs actual over time
  • Playbook: recommended interventions plus executive directives
  • Evidence: stage progression and factual rationale
  • Impact: affected wells, rig dates, governance flags, and local exposure facts

13) Chart System

Smart Alerts mixes two chart strategies:

  • 2D charts are rendered with Plotly through a dynamically imported client-only component
  • 3D-style column views are rendered by a local SVG engine called ExtrudedBarsChart

That design matters. It avoids relying entirely on heavyweight 3D libraries while still giving the user both flat and volumetric views of the same data.

Summary composition, driver stacks, and timeline divergence are all derived directly from the alert payload on the client.

14) Board-Level Metrics and Executive Framing

The board emits headline metrics such as Critical Wells, Avg Model Score, Delay Exposure, Rig Contagion, and Governance Flags. These are aggregated from the ranked alert set, not raw copied fields.

It also emits executive insights built from other tabs:

  • Portfolio pressure projects become board-level strategic context
  • Recent anomaly transitions become live risk-governance commentary
  • Data Integrity open exceptions affect whether leadership can trust the board as “real operations pressure” rather than bad data

That is the architectural reason Smart Alerts is valuable: it is where multiple intelligence layers are compressed into one intervention surface.

15) Why This Tab Creates Real Operational Value

In real operations, the hardest problem is not merely detecting that some wells are late. The hard problem is deciding what needs attention now, why it matters, who should act, and how confident the organization should be in that signal. Smart Alerts solves exactly that problem.

  • It converts noisy operational telemetry into a ranked intervention queue
  • It makes the score explainable with deterministic evidence and driver decomposition
  • It brings schedule, execution, risk, anomaly memory, and governance trust into one board
  • It preserves drill-down so the board is not just executive theater but an actionable operating surface

The tab is therefore best understood as a local decision engine for where field attention should move first.

Smart Alerts is a hybrid ranking system: part schedule analytics, part execution monitoring, part risk memory, and part governance trust layer. That combination is what lets it behave like an intervention board instead of a generic notification list.
