1) End-to-End Runtime Flow

SequelOntology has two jobs. First, it exposes the schema graph visually through the graph tab. Second, it serves as the retrieval substrate for Operations. The same graph that the user sees in 3D is also queried programmatically for semantic search, keyword search, and join-path resolution.

2) What SequelOntology Means in This Product

In Bashira, "ontology" does not mean an academic taxonomy disconnected from runtime. It means the product has converted the SQL schema into a machine-usable graph where structure, semantics, and joinability are explicit. A plain relational schema only tells you that columns exist. This ontology additionally tells you how those columns relate, which tables mirror each other, which columns are well keys, and where safe join paths exist.

That distinction matters because natural-language analytics fails when the system only knows table names. It succeeds when the system knows that WellMonitoringReport.pdo_well_id is semantically tied to Job_Progress_Report_GB.Well_ID, that certain tables mirror each other, and that some tables are direct or indirect references to the well hub.

3) Why a Graph Database Was Used

  • Schema relationships are first-class: joins, well references, and mirrored tables can be traversed directly.
  • Search is hybrid: same graph hosts fulltext and vector indexes over column meaning.
  • Explainability improves: retrieved columns and join paths can be shown as graph structure rather than hidden prompt text.
  • It scales better for reasoning: join-path discovery is natural in graph form.

6) How the Graph Is Built

  1. Seed constraints and indexes: uniqueness constraints for wells, tables, and composite column identity are created first.
  2. Create fulltext indexes: Neo4j fulltext search is enabled over column names and descriptions.
  3. Create a well hub: a central well entity is introduced so multiple tables can resolve toward the same business object.
  4. Load tables from CSV: each table becomes a Table node with description metadata.
  5. Load columns from CSV: each column becomes a Column node attached to its table via HAS_COLUMN.
  6. Mark well-key columns: columns like pdo_well_id or Well_ID get semantic flags.
  7. Create mirror links: related tables such as historical and latest variants are linked with MIRRORS.
  8. Create join hints: curated cross-table column mappings become JOINS_ON edges.
  9. Generate embeddings: MiniLM embeddings are computed for column descriptions and stored on Column nodes.
  10. Create Neo4j vector index: semantic search can now run natively inside Neo4j.

8) Why 384-Dimensional MiniLM Was Chosen

  • Compact: 384 dimensions are cheaper to store and query than larger embeddings.
  • Fast: suitable for local CPU use and fast startup.
  • Enough semantics: good for mapping business language to schema descriptions.
  • Operationally balanced: strong enough for schema retrieval without heavy GPU dependency.

9) Embedding and Search Mathematics

9.1 MiniLM embedding generation

Each column description is converted to a 384-dimensional dense vector. Operationally, the input text is a compact semantic descriptor such as column name plus description. The resulting vector is stored directly on the corresponding Neo4j Column node.

9.2 Vector search in Neo4j

During runtime, the user query is expanded, embedded, and compared against indexed column vectors. Neo4j returns the nearest columns using cosine similarity: cos(u,v) = (u.v) / (||u|| ||v||).

9.3 Fulltext search

The same question is also transformed into a Lucene-style keyword query and sent to Neo4j fulltext index. This is useful when wording contains exact IDs, rigid column names, or specific enterprise abbreviations.

9.4 BM25 lexical scoring

In parallel, an in-memory BM25 index over the column corpus scores how strongly each schema document matches query terms. Conceptually: score(D,Q)=sum(IDF(q_i) * (f(q_i,D)*(k1+1)) / (f(q_i,D)+k1*(1-b+b*|D|/avgdl))).

9.5 RRF fusion

The three ranked lists are merged using Reciprocal Rank Fusion: RRF(d)=sum(1/(k + rank_j(d))). This is important because no single retrieval mode is sufficient in enterprise schema search. Vector is strong on meaning, fulltext is strong on exact wording, and BM25 is strong on weighted lexical evidence.

10) Abbreviation Expansion and Domain NLP

The ontology layer is designed for oilfield and operations language, not generic English. Before retrieval, the query is expanded with a domain abbreviation map. This is why a question using terms like WMR, PDO, WBS, SCR, or PH can still reach the correct schema.

Abbreviation Expansion Intent Retrieval Benefit
WMR Well Monitoring Report Maps business shorthand to real table semantics
PDO Petroleum Development Oman / pdo_well_id Improves matching of well identity fields
WBS Work breakdown structure Improves project/task schema grounding
PH Project holder / PH name / PH employee ID Improves personnel/productivity mapping

11) Join Path Resolution

SequelOntology does not stop at returning a list of relevant columns. After top-ranked columns are selected, the system asks Neo4j for all JOINS_ON relationships touching those columns. This creates an explicit join-hint pack for the SQL generator.

  • Input: selected (table, column) pairs from fused retrieval.
  • Graph query: match column nodes, traverse JOINS_ON, return opposite side and description.
  • Output: structured join hints such as WellMonitoringReport.pdo_well_id JOINS_ON Job_Progress_Report_GB.Well_ID.
  • Value: SQL generation becomes less guess-based and more topology-aware.

12) How LangGraph Uses SequelOntology

The SequelOntology tab is visually separate, but technically it is embedded into the LangGraph analytics pipeline. Inside the graph workflow, the retrieve_schema node calls the RetrievalAgent, which in turn calls Neo4j hybrid search, BM25 search, RRF fusion, and join-path resolution. The final product of that step is a schema context string that the SQL agent consumes.

13) Frontend Rendering Path

The graph UI is rendered by the IntelligenceMatrix React component. It dynamically imports react-force-graph-3d to avoid SSR issues because WebGL must run client-side. On mount, it calls /api/knowledge-graph?limit=2000, normalizes nodes and edges, filters orphaned links, and feeds the result into ForceGraph3D.

16) Why This Is Hard Technically

  • Schema naming is inconsistent: similar concepts appear as Well_ID, well_id, and pdo_well_id.
  • Business meaning is not explicit in SQL: mirror tables and denormalized views need manual semantic encoding.
  • Enterprise abbreviations distort search: generic retrieval would miss domain shorthand without expansion logic.
  • Join correctness is fragile: LLMs often invent joins unless topology is grounded.
  • UI rendering is nontrivial: large graph scenes require filtering and camera control to remain readable.

17) Practical Limitation Notes

This ontology reflects the local clone and the seeded graph, not necessarily every object from the original client VPN environment. If the clone lacks a table, SequelOntology can still show meaningful topology for what exists locally, but it cannot invent missing metadata. The graph is therefore accurate for the current local deployment boundary, not for the inaccessible remote environment.

Also, the graph tab endpoint is optimized for visualization and limits returned nodes. The full reasoning graph used by retrieval remains richer than any single front-end view because the retrieval stack can query indexes and join paths programmatically beyond what is visible in one 3D frame.
