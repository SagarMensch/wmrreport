Working Of Modules

Portfolio - Technical Working Document

Detailed explanation of the Portfolio command surface: canonical operating frame generation from live wells, project-level aggregation, optional CPU ML enrichment for project delay probability and tail exposure, 2D and 3D Plotly views, and the agentic project feed.

Frontend: Next.js + React Command Surface: PortfolioCommand Proxy: /api/command-center?action=portfolio Backend: CommandCenterService Base Data: ForecastEngine latest snapshot Enrichment: CPU ML portfolio service on 8050
Primary Unit

Project

The tab lifts well-level execution state into project-level portfolio lanes so leadership can reason at a capital-allocation level.

Base Model

Hybrid

Uses live well progress and hybrid risk from ForecastEngine, then optionally augments projects with portfolio delay probability outputs.

View Style

2D + 3D

Treemap, scatter, composition donut, and topographic risk surface all render from the same filtered project-card set.

Decision Goal

Capital Pressure

The surface asks which projects are under execution pressure, which wells dominate that pressure, and what tail-risk exposure is implied.

1) Runtime Path

2) Frontend Shell

The frontend component is PortfolioCommand.tsx. It loads its data through the shared hook useCommandCenterData("portfolio"), which performs a fresh fetch("/api/command-center?action=portfolio") with cache: "no-store".

  • Built-in filters: all, active, inactive
  • Built-in chart tabs: topology, scatter, treemap, composition
  • Refresh action: Recalculate Risk triggers a refetch of the portfolio payload

3) Shared Command-Center Data Contract

The web proxy maps action=portfolio to backend route /api/command-center/portfolio. The envelope format is standardized:

  • status
  • data
  • optional error and detail fields

This matters because the Portfolio surface is one member of a larger command-center family. The shared hook and shared envelope keep all command surfaces consistent.

4) Canonical Operating Frame

The true backbone of the Portfolio surface is CommandCenterService._operating_frame(). This function materializes one well-level canonical table that every higher portfolio aggregate depends on.

Field Family Examples Source Logic
Identity project, well_id, well_name, cluster, rig_no, well_type Mostly from ForecastEngine latest snapshot; project is resolved through Job Progress category lookup when available
Progress and risk progress_pct, risk_score, risk_tier Progress from latest well snapshot; risk from ForecastEngine portfolio risk map or fallback tiering logic
Schedule state delay_days, rig_on_delay_days, rig_status Computed from expected versus actual rig-on and rig-off dates relative to today
Lifecycle stage completion loc_prep_pct, engineering_pct, construction_pct, commissioning_pct, flowline_pct, ohl_pct Read from well snapshot progress fields and normalized into percentage space
Event dates expected_rig_on, actual_rig_off, actual_start, flaf_issue, engineering_finish Copied into the operating frame to support downstream command surfaces and drilldowns

5) Project Lookup and Risk Map Enrichment

Two enrichers make the operating frame materially more useful than a raw well snapshot. The first is _project_lookup(engine), which uses Job Progress data to map each Well ID into a project category. Without this step, the portfolio would remain a flat list of wells with no rollup boundary.

The second is _risk_map(engine), which imports the hybrid risk scores computed by the forecasting layer. That means Portfolio is not inventing its own separate risk universe. It is consuming the same risk backbone used elsewhere in the system and then re-aggregating it at project scale.

  • _project_lookup(): converts wells into project groups
  • _risk_map(): attaches hybrid well risk to each row
  • Together they turn a well registry into a portfolio model

6) Project-Level Aggregation Logic

_build_portfolio_view() groups the canonical operating frame by project and then computes project cards. Each card is not a static record from SQL. It is a derived operating object built from multiple wells.

Metric How It Is Built
total_wells Count of wells in the grouped project
open_wells Rows not yet marked complete by progress threshold
completed_wells Rows with progress near 100 percent
delayed_wells Count of wells whose derived delay is positive
critical_wells Count of wells whose risk tier resolves to critical
avg_progress_pct Mean progress across all wells in the project
avg_risk_score / max_risk_score Mean and peak hybrid risk across wells
max_delay_days Worst derived delay value among project wells
clusters / lead_rig Distinct operational clustering metadata for the project

7) Optional CPU ML Enrichment from the Project Delay Service

Portfolio also supports a second layer of enrichment from http://127.0.0.1:8050/ml/portfolio/project-delay-probability. This is consumed by _project_delay_model_rows(). If that service is alive, each project card receives extra probabilistic fields. If it is down, Portfolio still works because these are additive, not foundational.

ML Field Meaning Why It Matters
avg_delay_probability_pct Average model-estimated chance of project delay Used in pressure ranking and portfolio summary cards
max_delay_probability_pct Peak delay probability inside the project Highlights the sharpest stress concentration
p95_delay_exposure_days Tail-risk delay estimate Useful for downside planning and escalation
expected_delay_days Mean modeled delay expectation Converts abstract risk into time exposure
risk_price Economic pricing of project risk Supports commercial or capital framing
nhpp_intensity Additional model-side pressure indicator Acts as a stress feature for comparison across projects

Architecturally this is important: Portfolio is not just a reporting tab. It is a hybrid decision surface that can combine deterministic well-state aggregation with optional probabilistic project-level inference.

8) Health Band, Pressure Ranking, and Summary Metrics

Once the project cards are built, the service assigns a health band through _health_band(score, delayed_wells, critical_wells). The logic is intentionally simple and explainable.

  • Critical: if there is at least one critical well, or at least three delayed wells, or score is at least 70
  • Watch: if there is any delay pressure, or score is at least 45
  • Stable: otherwise

The frontend then computes additional display metrics such as the overall pressure index. In PortfolioCommand.tsx, that index is derived as:

avg_delay_probability_pct * 0.62 + (projects_under_pressure / total_projects) * 38

This is a presentational portfolio stress indicator. It is not a raw database field. It is an executive compression of probability and breadth of pressure across the portfolio.

9) Why the Portfolio Summary Is Useful

A large operating portfolio is hard because local well facts do not automatically reveal where capital, rigs, schedule, and execution pressure are concentrating. Portfolio solves that by changing the level of abstraction.

  • It converts well-level operational telemetry into project-level capital and schedule objects
  • It ranks where intervention matters most instead of showing every well equally
  • It exposes concentration risk through delay, critical counts, and top-well extraction
  • It makes a large field readable for executive review in one surface

10) Plotly Views, 2D/3D Topology, and Visual Analytics

The Portfolio tab renders multiple visual forms through shared Plotly builders in DesertPlotly.ts. The point is not to decorate the screen. The point is to let the same project-card payload be interrogated from different analytical geometries.

View Builder What It Shows
Treemap buildTreemap() / buildTreemap3D() Relative project mass by portfolio metric such as risk, delay, or exposure
Scatter buildScatter() / buildScatter3D() Relationship between progress, risk, delay, and pressure across projects
Composition buildDonut() / buildDonut3D() Status or risk composition of the portfolio
Topology buildRiskSurface3D() A volumetric pressure surface for executive reading of relative portfolio stress

One important technical note: the 3D topology view is a derived visual surface, not a literal simulation of subsurface physics. It uses the project points as anchors and produces a readable stress landscape so the portfolio can be interpreted spatially. This should be understood as an analytical topology layer.

11) Agentic Feed: What It Really Is Today

The payload also returns an agentic_feed. In the current implementation this is generated by _generate_counterfactual_feed(project_cards) inside command_center_service.py.

That means the feed is currently a synthetic strategy layer created from the ranked project cards. It references concepts like P99 EVT risk, ST-GNN routing, or MARL coordination in the recommendation text, but it is not executing those models in the normal Portfolio API path.

There is a separate portfolio_agent.py in the repository containing a richer LangGraph-based synthetic execution officer, but the normal portfolio payload explored here is assembled directly inside the command-center service.

12) Caching, Freshness, and Response Characteristics

Portfolio has two opposing goals: remain live enough for operations, but not recompute every expensive aggregate on every paint. The design balances that with a backend cache and a frontend forced refresh strategy.

  • Frontend fetches with cache: "no-store"
  • Backend caches get_portfolio_view() for roughly 45 seconds
  • Refresh actions invalidate the UI view by re-requesting the backend payload
  • The canonical operating frame itself depends on refreshed ForecastEngine state

The result is a command surface that behaves live to the user while protecting the system from unnecessary recomputation.

13) Why This Tab Creates Real Value

Portfolio is where local execution data becomes executive operating intelligence. The tab does not merely show that some wells are late. It tells leadership which projects are absorbing schedule pressure, which clusters are accumulating critical wells, where probabilistic delay exposure is rising, and which individual wells are driving that portfolio stress.

Technically, its value comes from its layering:

  • well-level operational truth from the forecasting layer
  • project reconstruction through lookup and grouping
  • optional CPU ML enrichment for project delay probabilities
  • multi-geometry visual analytics through Plotly
  • executive synthesis via project cards, top wells, and strategy feed

In plain terms, this tab answers a hard management question: where should we intervene first, and why?

Portfolio is not a static reporting page. It is a project-scale decision layer built on top of the same hybrid risk backbone used by the broader Bashira operating system.
