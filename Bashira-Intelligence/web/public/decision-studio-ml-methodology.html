1. Scope of Decision Studio

Decision Studio is a predictive operations cockpit built on a live exported package. It is not one single model. It is a bundle of three related machine learning tasks plus survival analysis and explainability:

  1. Predicting future progress four weeks ahead.
  2. Predicting weeks remaining to completion.
  3. Predicting the probability of missing the target date.
  4. Estimating time-to-completion behavior with survival analysis.
  5. Explaining the learned behavior through weighted feature importance and SHAP outputs.
Important distinction. Decision Studio contains both model outputs and UI-derived summaries. For example, the risk score is a real model output. The progress distribution chart is a summary built from current scored wells. The rig radar view uses live rig aggregates and some synthetic presentation transforms for readability.

2. Input data and row grain

The training script loads five CSV inputs:

wmr_latest.csv Current portfolio snapshot used for final scoring and the live UI package.
plan_snapshot.csv Near-term plan fractions and target-end information.

2.1 Canonical well identifier

The canonical well key is pdo_well_id. This matters because a major earlier problem in Decision Studio was incorrect well counting and training on a reduced or distorted portfolio when the unique identifier was wrong.

2.2 Historical row grain

Historical training is based on well-week style snapshots. The pipeline explicitly deduplicates historical data by: 

dedupe key = (pdo_well_id, snapshot_date)

If multiple rows exist for the same well and snapshot, the pipeline keeps the row with the strongest completeness and the strongest progress information.

2.3 Current scoring row grain

Current scoring is one row per current well in wmr_latest.csv, enriched with lag features from history and joined plan, report, and SAP signals.

3. Feature engineering

The feature set combines live status, lags, phase progress, plan pressure, target pressure, and categorical operating context. The model candidate set contains 51 features: 44 numeric and 7 categorical.

3.1 Time-lag and momentum features

progress_lag1        = prior snapshot progress
progress_lag2        = two snapshots back
progress_lag4        = four snapshots back
progress_rolling3w   = rolling 3-snapshot mean
progress_velocity    = progress_t - progress_lag1
progress_accel       = progress_velocity - (progress_lag1 - progress_lag2)
remaining_to_complete = 1 - over_all_progress_percentages

3.2 Plan-pressure features

plan_5w_total  = W1 + W2 + W3 + W4 + W5 plan fractions
plan_curve_gap = over_all_progress_percentages - plan_cumcurrentmonthplanfrac

This lets the model see both near-term planned workload and how far the actual progress line sits above or below the plan curve.

3.3 Target-pressure features

days_to_target      = expected_target_date - snapshot_date
overdue_target_days = max(-days_to_target, 0)
is_overdue_target   = 1 if days_to_target < 0 else 0

3.4 Stall features

stall_1w_flag = 1 if abs(progress_velocity) < 0.005 else 0
stall_4w_flag = 1 if abs(overall_progress - progress_lag4) < 0.01 else 0

3.5 Phase-balance features

The pipeline uses phase columns such as location preparation, engineering, construction, commissioning, flowline, and OHL progress to build higher-order consistency signals: 

phase_mean_progress   = mean(phase columns)
phase_min_progress    = min(phase columns)
phase_max_progress    = max(phase columns)
phase_std_progress    = std(phase columns)
phase_alignment_gap   = overall_progress - phase_mean_progress

4. Target engineering

This is one of the most important parts of the pipeline. The models are not trained on arbitrary labels. Each target is created from forward-looking observed history.

4.1 Future progress target

future_progress_4w = over_all_progress_percentages shifted -4 snapshots within each pdo_well_id

This is the target behind the main progress model. It answers: given what we know at snapshot t, what progress level will we observe roughly four weekly snapshots later?

4.2 Completion event and observed completion date

is_complete = (over_all_progress_percentages >= 0.95) OR (actual_finish_date exists)

observed_completion_date =
  first future snapshot date where is_complete becomes true

4.3 Weeks remaining actual

weeks_remaining_actual = (observed_completion_date - snapshot_date) / 7

This becomes the target for the completion-timing ensemble.

4.4 Miss-target observed

eligible rows = expected_target_date is known
                AND expected_target_date <= latest snapshot in history

completion_or_censor = observed_completion_date if complete
                       else max historical snapshot date

miss_target_observed = 1 if completion_or_censor > expected_target_date else 0

This is the target for the risk classifier. It is not a hand-built score. It is an observed historical label saying whether the well ended up missing target or not.

5. Training and validation design

5.1 Time-based validation

The pipeline does not use random row shuffling. It performs a time split on snapshot dates: 

sorted unique snapshot dates
split index = max(1, int(number_of_dates * 0.8))
train rows  = snapshot_date < cutoff
valid rows  = snapshot_date >= cutoff

This is a much more defensible choice than in-sample evaluation because it tests the models on later time periods than those used for fitting.

5.2 Missing-value handling

  • Numeric features are converted with numeric coercion and filled with the training-frame median.
  • Categorical features are converted to string and missing values become Unknown.

5.3 GPU fallback logic

The Kaggle version tries GPU first for CatBoost and XGBoost. If GPU fails, it falls back to CPU. The current live package was successfully produced with GPU-backed training members.

7. Risk score, risk drivers, and tiering

7.1 Raw risk probability

Each risk model outputs the probability that a well will miss target. The ensemble first forms a weighted average of raw classifier probabilities.

7.2 Platt calibration

The raw risk probabilities are not used directly. They are calibrated with logistic regression applied to the logits of the raw ensemble output: 

clipped_p = clip(raw_probability, 1e-6, 1 - 1e-6)
logit_p   = log(clipped_p / (1 - clipped_p))
Platt calibration = LogisticRegression(logit_p, miss_target_observed)

This improves the probability scale so that the risk score behaves like a more usable calibrated probability.

7.3 Final risk score

risk_probability = calibrated_probability * 100
risk_score       = risk_probability
risk_tier        = thresholded(risk_score)
Key point. In the current pipeline, risk_score is not a separate handcrafted score. It is exactly the calibrated miss-target probability scaled to 0-100.

7.4 Tier thresholds

The thresholds are not hardcoded. They are derived from the distribution of calibrated validation probabilities. 

if enough unique values:
  run KMeans with 4 clusters on validation probabilities * 100
  sort the 4 cluster centers
  threshold = midpoint between neighboring centers
else:
  use empirical quantiles: 50%, 75%, 90%

7.5 What “risk drivers” means in this tab

The Decision Studio tab contains three different ideas that are easy to mix together:

  1. Risk score: calibrated probability of missing target.
  2. Predictive drivers: weighted feature importance from the progress ensemble.
  3. Risk SHAP drivers: signed SHAP effects from the CatBoost risk model, exported to shap_risk_drivers.png.

So “risk driver” can mean different things depending on which panel is being discussed. The portfolio-facing rank chart in the main tab is still the progress-model feature ranking, not the risk-classifier SHAP chart.

8. Completion dates, intervals, and survival analysis

8.1 Weeks remaining prediction

The weeks ensemble predicts weeks_remaining_predicted for each current well.

8.2 Completion dates

predicted_completion_date = snapshot_date + round(weeks_remaining_predicted * 7 days)

completion_date_early =
  snapshot_date + round(max(weeks_remaining_predicted + residual_q10, 0) * 7 days)

completion_date_late =
  snapshot_date + round(max(weeks_remaining_predicted + residual_q90, 0) * 7 days)

The early and late dates come from the 10th and 90th quantiles of validation residuals in the weeks-to-completion model. In practice, this is what produces the timing interval shown in the forecast chart.

8.3 Survival model

The pipeline also fits a Cox proportional hazards model on one row per well. This is separate from the weeks ensemble. Its purpose is ranking and completion-timing behavior analysis, not replacing the point estimate.

Survival training row construction

start_date = first historical snapshot for a well
end_date   = observed completion date if complete, else last observed snapshot
duration_weeks = max((end_date - start_date)/7, 0.1)
event_observed = 1 if completed, else 0

Cox feature selection and stabilization

The survival block prunes features with too little variance or extreme collinearity, then retries Cox fitting with stronger penalization if needed. This was added specifically so the live package would converge reliably.

9. Feature importance, SHAP, and completion drivers

9.1 Weighted ensemble feature importance

The main feature-importance ranking is not permutation importance. It is a weighted average of model-native feature importances across the progress ensemble: 

for each progress ensemble member:
  raw_importance = model-native importance
  normalized_importance = raw_importance / sum(raw_importance)
  weighted_component = normalized_importance * member_weight

final_importance(feature) = sum(weighted_component across members)
normalize final_importance to sum to 1

9.2 SHAP for the progress model

The live package exports shap_beeswarm.png and shap_importance.png from the first CatBoost member in the progress ensemble, using a sampled validation frame of up to 400 rows.

9.3 SHAP for the risk model

The live package exports shap_risk_drivers.png from the first CatBoost member in the risk ensemble. This is the signed average SHAP direction chart for the miss-target classifier.

9.4 Cox completion-driver chart

The completion-driver chart in the tab is the Cox hazard-ratio output from the survival model. It is not SHAP. It answers which covariates are associated with faster or slower completion in the survival formulation.

10. How each panel in the Decision Studio tab is calculated

KPI cards

Portfolio = row count of risk_scores.csv.

Critical / At Risk / Watch / Healthy = counts grouped by risk_tier.

Risk Distribution

Donut or 3D tier pillars built from grouped counts of risk_tier in risk_scores.csv.

Progress Distribution

Uses current well progress percentages and buckets them into 10-point bands: 0-10, 10-20, ..., 90-100.

Rig Capability Radar

Built from live rig aggregates. For each rig, the UI computes: 

progress = average current progress across wells on the rig
load     = min(well_count * 4, 100)
eff      = min(progress * 1.2, 100)

This is a presentation summary, not a separately trained ML model.

Rig Fleet Performance

Grouped by rig_no from live scored rows: 

avg_progress = mean(over_all_progress_percentages)
wells        = nunique(pdo_well_id)
mean_risk    = mean(risk_score)

Well table

Comes from priority_wells_final.csv, sorted by: 

sort by risk_score desc,
        weeks_remaining_predicted desc,
        current_progress_pct asc

Predictive Drivers

Uses feature_importance.csv. In 2D it is a horizontal bar chart; in 3D it is the same ranking rendered as 3D prism bars.

Driver Distribution

Uses shap_beeswarm.png from the progress model.

Completion Probability

Uses survival_kaplan_meier.png. This is Kaplan-Meier probability of wells still being incomplete over time, grouped by rig-tier cohorts.

Timeline Forecast

Uses completion_forecast.png. It takes the earliest 20 predicted completion rows and shows: 

horizontal line = early to late completion window
red point       = median completion estimate
vertical zero   = reference snapshot date

Model Accuracy

Uses ag_metrics.json and training_manifest.json. These values are not manually typed in the UI. They are loaded from the live generated package.

Completion Drivers

Uses survival_cox_hazard.png. This is the Cox hazard-ratio chart from the survival block, not a generic curve and not a copied image from an older package.

Final reading guidance. If someone challenges Decision Studio, answer in this order:
  1. The package is trained on deduped historical well snapshots keyed by pdo_well_id.
  2. The main risk score is calibrated miss-target probability times 100.
  3. The tier thresholds are learned from validation probability structure, not hardcoded.
  4. The forecast dates come from the completion ensemble plus residual quantile bands.
  5. The explainability outputs come from weighted feature importance, SHAP, and a separate Cox survival model.
