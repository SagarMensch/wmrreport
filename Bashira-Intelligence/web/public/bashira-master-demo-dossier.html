Core positioning

Bashira Intelligence is not just a BI layer. It is an execution intelligence operating system with three stacked advantages: retrieval depth, operational ML, and intervention-oriented user surfaces. The product does not stop at describing what happened. It attempts to answer four progressively harder questions:

  1. What is happening right now across wells, rigs, stages, and projects?
  2. Where is risk accumulating and why?
  3. What action should be taken next, by whom, and with what expected effect?
  4. What would likely happen under a counterfactual change in the operating plan?

That stack is what separates a true operating platform from a dashboard that only decorates the data.

1. Executive thesis

Bashira Intelligence is designed to look and behave like institutional software, but its real advantage is under the surface. The platform combines live SQL access, schema-aware retrieval, a knowledge graph, an explainable SQL agent, forecasting, CPU-based portfolio risk, causal analysis, integrity controls, and multiple command surfaces that convert model output into operational decisions.

What the product already proves

  • It can run on the local offline SQL Server clone without the client VPN.
  • It can survive schema mismatch using local-fallback logic instead of failing closed.
  • It exposes well-level, rig-level, stage-level, and portfolio-level intelligence.
  • It has both descriptive and prescriptive surfaces: forecast, alerts, playbooks, causal actions.
  • It has operational governance, not just analytics, through the integrity engine and data dictionary.

What this means in a client evaluation

  • The product can be positioned as an operating system, not a point dashboard.
  • The ML is tied to real workflow objects: wells, rigs, delays, stages, exceptions, and directives.
  • The UI can present both premium executive polish and technical depth for scrutiny.
  • Every strong demo moment can be backed by a logic narrative rather than visual theater alone.
  • Compared with a typical POC, Bashira has more explainability, more fallback resilience, and more operational breadth.

3. Platform architecture

The architecture is layered so the product can continue working even when one source is degraded. This matters in the current local demo setup because the product is not operating against the original VPN-backed client source of truth. Bashira remains useful because it is intentionally redundant.

Layer 1

Truth sources

Local SQL Server clone, CSV fallback assets, job progress export, well monitoring history, and supporting plan and productivity tables.

Layer 2

Retrieval context

BM25 on columns, Neo4j graph context, SQL schema introspection, and entity-aware prompt assembly.

Layer 3

Reasoning engines

Intent, retrieval, SQL generation, validation, execution, charting, forecast logic, anomaly layers, causal command, and CPU action simulation.

Layer 4

Operating surfaces

Portfolio, alerts, rig command, field atlas, integrity, causal, predictive, decision, and governance surfaces for different user roles.

Agent pipeline used by the conversational and decision layer

Intent -> Retrieval -> SQL -> Validation / Execution -> Chart

The orchestrator tracks each step as a reasoning step. That makes the system both inspectable and demoable: the product can expose not only a result but the chain of reasoning that produced it.

Retrieval parameters

  • VECTOR_TOP_K = 40
  • BM25_TOP_K = 40
  • FULLTEXT_TOP_K = 40
  • FINAL_TOP_K = 75
  • RRF_K = 60

These settings bias toward high-context retrieval, which is useful in a complex schema where a single question may touch multiple operational entities.

Frontend and visualization stack

  • Next.js 15.5.14 and React 19.1
  • Tailwind v4 for design tokens and layout
  • Plotly for analytical 2D and 3D charting
  • Three.js and react-three-fiber for spatial or immersive views
  • Framer Motion for transitions and UI staging
  • Recharts and Force Graph where lightweight or topology views are useful

5. Risk and forecast logic

The platform does not rely on one monolithic score. Different engines are responsible for different kinds of risk, and that is exactly the right design. Forecast risk, portfolio risk, rig pressure, and causal uplift are related but not identical concepts.

Forecast engine base risk logic

In the forecast engine, the detailed well risk score is built from three interpretable components:

  • risk_prog = (1 - progress) * 35
  • risk_vel = max(0, (0.05 - velocity) / 0.05) * 25 when velocity is below 0.05
  • risk_stall = 20 when velocity is non-positive and progress is below 0.9

The total is clamped to a 0 to 100 band. This is intentionally interpretable: progress drag, velocity drag, and stall penalties are visible and explainable.

Forecast tiering

  • Critical score 75 or higher
  • High risk score 55 to 74.9
  • Watch score 35 to 54.9
  • Healthy below 35

Alternative model banding

The product also uses a slightly different banding in the broader portfolio map:

  • 70+ Critical
  • 40+ High risk
  • 20+ Watch
  • Below 20 Healthy
Portfolio risk components currently used in the local hybrid map
Component Meaning Why it matters
schedule_variance_risk Difference between planned and actual progress timing Captures slippage against the intended sequence.
overdue_deadline_risk Penalty for expected dates that have already passed Turns latent schedule miss into explicit risk.
stagnation_risk Penalty for flat or declining recent throughput Detects slow motion failure before total stall.
readiness_risk Stage readiness and slowest-stage gap pressure Flags bottlenecks before drilling windows are missed.
drilling_execution_risk Rig-on and rig-off delay, drilling sequence drag Connects well schedule issues to actual rig execution.
governance_risk Data quality and completeness drag on operational confidence Prevents false certainty from weak source records.

Why this is stronger than a simple delay score

Many tools claim to be predictive but simply repaint schedule delay in a colored band. Bashira separates progress drag, velocity drag, stage drag, readiness drag, and governance drag. That decomposition is important because it means the product can not only say that a well is risky, but also show which mechanism is causing the risk and which action family is most likely to reduce it.

6. Command and operations logic

The command-center layer is where the product starts to behave like an operating system. These surfaces do more than summarize data. They rank, cluster, prioritize, and attach interventions.

Smart Alerts logic

  • Blends local rig delay exposure, weekly plan divergence, stage stall, portfolio risk prior, and data confidence.
  • Ranks alerts by model score and intervention urgency.
  • Attaches evidence, system flags, and next-step playbook guidance.
  • Connects individual alert severity to portfolio contagion instead of treating each issue as isolated.

Rig Operations logic

  • Uses schedule slip, weekly divergence, stage bottlenecks, rig queue contagion, and CPU Isolation Forest outlier signals.
  • Rolls well pressure into rig pressure and rig recovery confidence.
  • Highlights bottleneck ownership and next directive family.
  • Supports both 2D and 3D rig pressure visual inspection.

Field Atlas logic

Spatial clustering, hotspot cells, corridor pressure, neighborhood density, and zone-level exposure mapping convert separate wells into a geographic command view.

Location Prep logic

Stage readiness logic identifies the slowest stage, the widest gap, and the most exposed transition between readiness and drilling windows.

Watchlist logic

Watchlist consolidates the most important objects to follow across multiple engines: forecast, rig command, alerts, and integrity.

7. Causal and counterfactual layer

The causal layer is one of the strongest differentiators in the product because it turns execution data into lever-based reasoning. Instead of only asking which wells are bad, the user can ask which actions or operating changes are likely to shift an outcome.

Causal workspace construction

  • Loads tables into a normalized causal workspace.
  • Builds feature datasets across job progress, task daily, ATP, SAP drilling, productivity, and supporting tables.
  • Computes join coverage and dataset quality signals.
  • Applies local fallback when a client-only table is missing, for example deriving plan snapshot features from job progress reports.

Bayesian and fast-path execution

  • Runs a fast CPU layer for immediate command responsiveness.
  • Refreshes the Bayesian layer asynchronously.
  • Uses CmdStan MCMC when the Windows toolchain is available.
  • Falls back to Laplace approximation when MCMC compilation or runtime is unavailable.

Why the causal layer matters in a demo

Many products can rank a delay. Far fewer can articulate what change in the operating plan would most likely reduce that delay and how confident the system is in that claim. Bashira can speak that language. Even in the local offline environment, it can still construct a causal workspace, report coverage, and return counterfactual reasoning.

8. Data integrity engine

Enterprise trust requires more than model outputs. The integrity engine is the control layer that tells the user whether the underlying operational records are sound enough to support downstream decision making.

Current five live integrity rules
Rule Source What it checks Current severity
TD-01 Required Quantity Exceeded task_daily Required quantity is greater than captured daily quantity on linked rows. High
TD-02 Daily Execution Fields Missing task_daily Quantity, employees, equipment, hours, or PH name is blank, null, or invalid. Critical
TD-03 Completed Without Actual End task_daily Completed flag is true while actual_end is missing. Critical
ATP-01 Actual Duration Exceeds Target ActivityTaskPlan Actual duration exceeds target duration. Medium
ATP-02 Actual Quantity Exceeds Forecast ActivityTaskPlan Actual quantity is greater than forecast quantity. Medium

Local sample operational mode

In the client wording, several task_daily rules are described as URL-linked checks. In the local offline clone there are no URL-linked rows, so the service transparently switches into a sample operational mode. In that mode, the applicable basis becomes:

ActionOn + task_code + project_id

This keeps the rule engine live and honest instead of pretending the client-only URL-linked context still exists locally.

Current live exception picture

The local environment currently surfaces 30 open TD-02 exceptions. Those rows are not abstract warnings; each record contains rule id, severity, record key, summary, failing fields, expectation, and fix path, plus a direct SSMS evidence view.

10. Positioning versus typical POC dashboards

This section is intentionally written for internal use. It frames the product against the kind of competitive proof-of-concept that often wins initial attention with polished screens but shallow operating depth.

What a typical POC dashboard gets wrong

  • It shows risk color without exposing the mechanics behind that score.
  • It over-emphasizes status cards and under-emphasizes action pathways.
  • It breaks when the exact client schema is not available.
  • It treats data quality as a side note instead of a first-class operating risk.
  • It lacks a counterfactual layer, so it can only describe, not reason about change.

How Bashira should be positioned instead

  • As a command operating system rather than a static dashboard.
  • As an explainable ML and reasoning platform, not only a visualization layer.
  • As resilient to imperfect source conditions through local fallback paths and governed approximations.
  • As stronger on intervention language: what to do next, why, and where the payoff is likely to come from.
  • As more trustworthy because it includes integrity, dictionary, reasoning traces, and diagnostics.

11. Demo narrative and talking points

The strongest demo is not a guided UI tour. It is a sequence that progressively proves depth. The goal is to show that Bashira is not merely reporting the portfolio. It is actively structuring operating decisions.

Act I: Establish truth and control

  1. Open Portfolio or Predictive Studio and show the live local well count.
  2. Explain that the environment is running from a local SQL clone, not a mocked dashboard.
  3. Show that the system can still expose structured logic despite offline constraints.

Act II: Surface risk and action

  1. Move into Smart Alerts or Rig Operations.
  2. Select one critical asset and walk through the drivers, stage gap, and action playbook.
  3. Explain that the score is decomposed rather than painted.

Act III: Prove governance and depth

  1. Open Data Integrity and show a real TD-02 record with failing fields.
  2. Open Causal Command and explain counterfactual levers.
  3. Close by positioning Bashira as a system of command, not an isolated dashboard.

Suggested one-line positioning for the client video

"Bashira is an institutional operating system for field execution. It does not stop at status visualization. It quantifies risk, traces root cause, validates the source data, and recommends interventions at well, rig, stage, and portfolio level."
