1. The Short Answer

The current Decision Studio risk score is a calibrated machine-learned probability that a well will miss its target completion date, converted to a 0-100 scale and then bucketed into four tiers.

raw model probability -> Platt-calibrated probability -> multiply by 100 -> round to 1 decimal -> classify tier

In the current package, risk_score is numerically identical to risk_probability. The difference is semantic:

risk_probability

The calibrated model probability, expressed as a percentage from 0.0 to 100.0.

risk_score

The same percentage value, exposed as the operational score shown in the UI.

risk_tier

The score bucket derived from learned thresholds: HEALTHY, WATCH, HIGH_RISK, or CRITICAL.

2. What the Risk Score Is Predicting

The score does not directly predict future progress. It predicts whether a historical well snapshot eventually missed its target date. The binary training label is miss_target_observed.

Business meaning A score of 92.7 means the model believes the well belongs in a historically very high miss-target-risk pattern, after calibration against real observed historical outcomes.
Important distinction Decision Studio also trains a 4-week future progress model and a weeks-to-completion model. Those are adjacent model products. The risk score comes from the binary miss-target classifier, not from those two regression heads.

4. How the Historical Training Frame Is Built

The trainer first constructs a single historical well frame by normalizing WMR, then joining the latest plan, report, and SAP aggregates onto each historical snapshot row. It then deduplicates to one row per pdo_well_id + snapshot_date, keeping the most complete row.

history = normalize_wmr(wmr_full) + latest plan join + report join + SAP aggregate join - duplicate well-date rows

6. How the Binary Risk Label Is Created

The label used for the risk classifier is not typed manually. It is built from historical target dates and observed completion behavior.

6.1 Target date construction

The trainer first creates expected_target_date using this priority:

expected_target_date = exp_rig_off_location_sap_data else plan_latest_target_end

6.2 Completion state construction

A snapshot is treated as complete if either of these is true:

is_complete = (over_all_progress_percentages >= 0.95) OR (actual_finish_date is not null)

For each historical snapshot, the trainer finds the first future snapshot where the well becomes complete and stores it as observed_completion_date.

6.3 Eligibility for risk labeling

A historical snapshot only gets a risk label if the target date is already observable inside the available history window.

eligible = expected_target_date is not null AND expected_target_date <= max_snapshot_date_in_history

6.4 Final binary label

If a well completed after target, or had still not completed by the end of the history window after the target had already passed, it is labeled as a miss. 

completion_or_censor = observed_completion_date if present else max_snapshot_date

miss_target_observed =
    1 if eligible and completion_or_censor > expected_target_date
    0 if eligible and completion_or_censor <= expected_target_date
    NaN if not eligible
Why this matters The model is not trained on guesses. It is trained on historical snapshots where there is enough later history to know whether the well did or did not miss the target.

7. Time-Based Validation Split

The trainer does not randomly split rows. It uses a time-based validation split on distinct snapshot_date values. This matters because random splits can leak future structure into training.

sort unique snapshot dates cutoff = 80th percentile date boundary train = rows before cutoff valid = rows on or after cutoff

This means the risk score is validated out-of-time, which is much closer to the real operational use case: score current wells based on what earlier historical wells taught the system.

9. Missing-Value and Category Handling

CatBoost path

Numeric features are converted to numbers and filled with each column median. Categorical features are filled as "Unknown". CatBoost then learns directly on the mixed-type frame.

XGBoost path

Categorical values are label-encoded from the training frame. Unknown categories map to -1. Numeric features are filled with the training median for that column.

Why this matters The production score can still be generated when some current fields are missing. Missingness does not break the score path. It becomes part of the learned pattern through encoding and fill rules.

10. How the Ensemble Probability Is Formed

Each base risk model produces a probability on the validation fold. The trainer then computes one weighted average. 

risk_weights = normalize_weights(validation_auc_scores, higher_is_better=True)
raw_ensemble_probability = weighted_average(member_probabilities, risk_weights)

The current weight formula is:

shifted = clip(auc_i - min_auc + 1e-6, 1e-6, None)
weight_i = shifted / shifted.sum()

This means the better the validation AUC, the more that member influences the final raw probability. In the current run, the two CatBoost members dominate the risk score because they materially outperform the XGBoost classifier on the validation split.

11. Platt Calibration

The raw ensemble probability is not used directly. It is first calibrated with a logistic model trained on validation data. This is important because raw classifier probabilities are often overconfident or poorly scaled.

11.1 Calibration input

The trainer takes the raw weighted ensemble validation probabilities, clips them into a safe range, converts them into logits, and fits a one-dimensional logistic regression: 

clipped = clip(raw_probability, 1e-6, 1 - 1e-6)
logit = log(clipped / (1 - clipped))
platt_model = LogisticRegression(max_iter=1000)
platt_model.fit(logit, observed_labels)

11.2 Calibration output

For both validation and current wells, the same transform is applied: 

risk_prob = clip(platt_model.predict_proba(logit(raw_ensemble_probability))[:, 1], 0, 1)
Meaning Calibration is what turns a strong classifier into a more defensible operational probability scale.

12. How the Tier Thresholds Are Learned

The system does not use manually typed thresholds like 25, 50, and 75. It learns thresholds from the calibrated validation score distribution.

12.1 KMeans thresholding

If there are at least 4 distinct calibrated values, the trainer clusters the 0-100 validation scores into 4 groups:

values = calibrated_validation_probability * 100
kmeans = KMeans(n_clusters=4, n_init=10, random_state=42)
kmeans.fit(values.reshape(-1, 1))
centers = sort(cluster_centers)
threshold_1 = midpoint(center_1, center_2)
threshold_2 = midpoint(center_2, center_3)
threshold_3 = midpoint(center_3, center_4)

13. How Current Wells Are Scored

After training, the model constructs a current well frame from wmr_latest.csv, merged with the latest lag features, plan fields, report fields, and SAP fields. That current frame is then passed through the three trained model families. 

progress_4w_pred = progress ensemble(current features)
weeks_remaining_predicted = weeks ensemble(current features)
risk_raw = risk ensemble(current features, proba=True)
risk_prob = calibrator.transform(risk_raw)

risk_probability = round(risk_prob * 100, 1)
risk_score = risk_probability
risk_tier = classify_tier(risk_score, thresholds)
Critical point There is no extra hidden formula after calibration. The current live score is exactly the calibrated probability percentage.

16. What This Score Is Not

  • It is not a manually weighted KPI index.
  • It is not just a function of days_to_target.
  • It is not the same thing as predicted progress.
  • It is not a causal proof that one variable mechanically caused the miss.
  • It is not based on static business thresholds typed by hand.
The safest sentence to use The Decision Studio risk score is a calibrated, out-of-time validated probability that the current well snapshot belongs to a historical pattern that missed target completion.

19. Final Plain-English Summary

The Decision Studio risk score is built by training three historical classifiers to recognize whether a well snapshot eventually missed its target date. Those classifiers read a rich operational feature set covering progress history, plan alignment, phase readiness, rig context, SAP load, and target pressure. Their validation probabilities are blended, then calibrated using Platt scaling so the probability scale is more trustworthy. That calibrated probability is converted into a 0-100 score, and the score is then assigned to a learned tier using thresholds discovered from the validation distribution itself. In the current live package, the final risk score is therefore a calibrated miss-target probability percentage, not a hand-built weighted KPI.
