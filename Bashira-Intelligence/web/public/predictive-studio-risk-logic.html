Predictive Studio Governance Note

How Predictive Studio Calculates Risk, Uses Live SSMS Data, and Handles Forecast Logic

This document explains the current Predictive Studio implementation in plain business language and code-accurate detail. It is designed so management, product, engineering, and client stakeholders can all understand what is live fact, what is derived logic, what is secondary ML evidence, and why certain fields such as commissioning are deliberately filtered instead of shown blindly.

Operating Principles

Predictive Studio now follows a governed hierarchy: live fact first, cleaned historical evidence second, deterministic score third, and machine learning only as secondary signal. This is the core design change that makes the screen explainable and auditable.

Principle 1

Facts must come from live SSMS

Header cards, phase cards, and milestone dates should reflect the latest operational row, not an old CSV export if live SQL is available.

Principle 2

Risk must be explainable

The official risk score is the sum of named, auditable components with fixed caps and clear business meaning.

Principle 3

Untrustworthy fields must be filtered

If a source field is clearly defaulted or contradictory, the UI should show N/A or a placeholder state instead of pretending certainty.

Principle 4

ML is supporting evidence only

The CPU ML signal is preserved as an early-warning indicator, but it does not currently control the official headline risk score.

Management reading: the number in the Risk Assessment card is intended to be a governed operational score. It is not a hidden model output.

Data Lineage in Predictive Studio

Each well detail page now carries source lineage. That means the UI can explicitly say where the current facts, historical series, forecast line, and risk logic came from.

Current State

Official

Current-state data comes from the latest WellMonitoringReport row for the selected well, ordered by Week_Number descending.

If live SQL is unavailable, the engine falls back to the cached latest snapshot file. Live SQL is always preferred.

History

Official

Historical progress comes from WMR_Full, not from the latest snapshot table. The history is cleaned before plotting to remove duplicate same-day rows and inconsistent backward movement.

Forecast

Secondary

If the CPU ML service on port 8050 is available, the forecast line is returned from /ml/forecast/{well_id}. If it is not available, the engine uses a transparent 4-week trend fallback computed from the cleaned live history.

Risk

Official

The headline risk score is labeled internally as institutional_live_score. It is deterministic and built from live schedule, readiness, stagnation, drilling, and governance signals.

History cleaning steps used before plotting the trajectory Current implementation
  1. Normalize all date fields and progress fields into consistent types.
  2. Drop rows where Week_Number is null.
  3. Normalize Week_Number to date only, so duplicate timestamps on the same day collapse cleanly.
  4. Group by week and take max for numeric progress fields on that date.
  5. Apply cummax() across the ordered history so cumulative progress does not move backward from data noise.
  6. Replace old or inconsistent historical well names with the latest canonical well name from the current-state row.

Official Risk Score Formula

The official risk score is a deterministic composite from six operational components. Each component is capped, added to a total score, and then mapped to a risk tier. Machine learning does not currently change this official score.

Important: the older stall_probability model is still computed in the CPU ML service, but it is now shown only as an advisory early-warning signal. It is not the official management risk score.

Tier Mapping

How the final score becomes a tier

  • CRITICAL: score >= 70
  • HIGH_RISK: score >= 40 and < 70
  • WATCH: score >= 20 and < 40
  • HEALTHY: score < 20
Official composite

Final score equation

official_risk_score = min(100, schedule_variance_risk + overdue_deadline_risk + stagnation_risk + readiness_risk + drilling_execution_risk + governance_risk)

The value is capped between 0 and 100, rounded to one decimal place, then mapped to a tier.

Detailed interpretation of each component Drill down

1. Schedule Variance

This component only applies when the system knows both an actual start and an expected rig-off date, and no actual rig-off has been recorded yet. It converts elapsed time into an expected progress curve, then measures how far the live progress is behind that curve.

2. Overdue Deadline

This component activates when the expected rig-off date has passed with no actual rig-off, or when the rig-off date is close and the well still has low completion. It is the main deadline-pressure component.

3. Execution Stagnation

This component requires usable history. If the last four cleaned history points have moved by less than one progress point, the well is treated as stalled operationally even if it is not yet formally overdue.

4. Readiness

This is the controlled treatment for 0% progress wells. A not-started well is not automatically critical. Its score depends on whether it has already started in some form and how near the target deadline is.

5. Drilling Execution

If the well is already marked ROL and overall completion is still below 60%, the score adds execution pressure because the well is in an active drilling state without enough total progress.

6. Governance

If the latest row shows an MOC has been raised but not approved, the score adds a governance penalty. This is intentionally small but visible because it signals control risk rather than pure schedule risk.

Why the CPU ML stall probability is not the headline score Secondary evidence only

The CPU ML service trains a LightGBM model on a target called is_stalled. That target is defined as:

  • progress > 0
  • progress < 0.95
  • latest history point older than 14 days

The model currently uses these features:

  • progress
  • rig_efficiency
  • cluster_density
  • momentum

Its output is stall_probability_pct. That output is now treated as an advisory early-warning signal only. It is preserved in risk attribution, but it does not change the official headline score.

Commissioning Logic and the 100% Placeholder Problem

The current SSMS snapshot shows a structural anomaly: the field overall_comm_progress_100 is 1.0 for all latest rows. That cannot be trusted as proof that commissioning really happened for every well, because the supporting commissioning dates are often null.

What changed in the UI: Predictive Studio no longer blindly renders commissioning as completed just because overall_comm_progress_100 = 1.0. If commissioning start and finish are both null and the well is not fully complete overall, the screen now treats commissioning as a placeholder and renders it as N/A in the cards.
Exact placeholder rule Current safeguard

Commissioning is treated as a placeholder when all of the below are true:

  • overall_comm_progress_100 >= 100%
  • actual_comm._start_date IS NULL
  • actual_comm._finish_date_with_in_2_days_from_actual_engg._completion_date IS NULL
  • over_all_progress_percentages < 100% or null

If that rule is triggered:

  • the top commissioning card shows N/A instead of 100%
  • the phase execution panel treats commissioning as NOT_STARTED unless real dates exist
  • the raw source value is still preserved internally as comm_progress_raw for audit
Why this matters

Client-safe explanation

The UI is not inventing 100% commissioning. The source field itself is populated as 1.0 across the latest snapshot. The UI now protects against showing that value blindly when the supporting commissioning dates do not exist.

In other words: the raw source value is real, but its business meaning is not reliable enough to expose directly.

Forecast and Trajectory Logic

The trajectory chart on the well detail page is built from cleaned live history. The forecast overlay is supplied by the CPU ML service if available, otherwise the engine falls back to a transparent short-horizon trend model.

Actual line

Cleaned live history

The plotted actual line uses the cleaned WMR_Full weekly history described earlier.

Forecast line

CPU ML first

If backend 2 responds on 127.0.0.1:8050, the screen uses the CPU ML forecast response for the selected well.

Fallback

Heuristic but explicit

If the CPU ML service is unavailable, the engine uses a slope-based 4-week forecast on cleaned recent history and labels the lineage accordingly.

Fallback forecast formula Used only when CPU ML is unavailable
  1. Take the last up to 14 cleaned history points.
  2. Fit a linear trend on progress percentage over day index using numpy.polyfit.
  3. Clamp negative slope to zero, because negative completion is not meaningful.
  4. Project 4 weekly points forward.
  5. Build a confidence band using 1.96 * residual_std * sqrt(horizon).
  6. If the slope is materially positive, estimate completion date and weeks remaining.

SQL Proof You Can Show the Client

All SQL below is read-only and safe to run against the current environment. These queries prove that the commissioning anomaly comes from live SSMS data, not from the UI layer.

1. Single well proof for 25115 / RASW_GH_WI_LOC5
SELECT TOP 1
    well_name_after_spud,
    pdo_well_id,
    TRY_CONVERT(date, Week_Number) AS snapshot_date,
    TRY_CAST([over_all_progress_percentages] AS float) AS overall_progress,
    TRY_CAST([overall_comm_progress_100] AS float) AS commissioning_progress,
    [actual_comm._start_date] AS commissioning_start,
    [actual_comm._finish_date_with_in_2_days_from_actual_engg._completion_date] AS commissioning_finish
FROM WellMonitoringReport
WHERE TRY_CONVERT(nvarchar(50), pdo_well_id) = '25115'
ORDER BY TRY_CONVERT(date, Week_Number) DESC;
2. Portfolio-wide proof that latest commissioning progress is defaulted to 1.0
SELECT
    COUNT(*) AS total_wells,
    SUM(CASE WHEN TRY_CAST([overall_comm_progress_100] AS float) = 1 THEN 1 ELSE 0 END) AS commissioning_100,
    SUM(CASE WHEN TRY_CAST([overall_comm_progress_100] AS float) = 0 THEN 1 ELSE 0 END) AS commissioning_0,
    SUM(CASE WHEN [actual_comm._start_date] IS NOT NULL THEN 1 ELSE 0 END) AS commissioning_started_rows,
    SUM(CASE WHEN [actual_comm._finish_date_with_in_2_days_from_actual_engg._completion_date] IS NOT NULL THEN 1 ELSE 0 END) AS commissioning_finished_rows
FROM WellMonitoringReport;
3. Show live risk-input fields used by the institutional score for a sample well
SELECT TOP 1
    well_name_after_spud,
    pdo_well_id,
    TRY_CONVERT(date, Week_Number) AS snapshot_date,
    TRY_CAST([over_all_progress_percentages] AS float) AS overall_progress,
    TRY_CONVERT(date, [actual_start_date]) AS loc_prep_start,
    TRY_CONVERT(date, [engineering_actual_start_date]) AS engineering_start,
    TRY_CONVERT(date, [const._actual_start_date]) AS construction_start,
    TRY_CONVERT(date, [actual_rig_on_date]) AS rig_on,
    TRY_CONVERT(date, [actual_rig_off_date]) AS rig_off,
    TRY_CONVERT(date, [exp.rig_off_location_sap_data]) AS expected_rig_off,
    [buffer_status],
    [moc_raised],
    [moc_approved]
FROM WellMonitoringReport
WHERE TRY_CONVERT(nvarchar(50), pdo_well_id) = '25115'
ORDER BY TRY_CONVERT(date, Week_Number) DESC;
4. Show the history used for stagnation logic
SELECT
    TRY_CONVERT(date, Week_Number) AS snapshot_date,
    TRY_CAST([over_all_progress_percentages] AS float) AS overall_progress
FROM WMR_Full
WHERE TRY_CONVERT(nvarchar(50), pdo_well_id) = '25115'
ORDER BY TRY_CONVERT(date, Week_Number) ASC;

What the Screen Now Means

UI element What it means now What it does not mean
Risk ring beside Risk Assessment The official deterministic risk score for the selected well. It is no longer a disguised progress ring.
Commissioning phase card Commissioning progress only when source evidence is credible. It does not blindly trust overall_comm_progress_100.
Risk Attribution panel The named score components that actually contributed to the official score. It is not a generic AI explanation panel.
Forecast lineage label Whether the chart is showing CPU ML forecast or fallback forecast. It does not imply that every line is model-generated all the time.

Current Limitations

  • The official score is now auditable, but it is still a rules-based operational score. It is not yet a formally client-approved enterprise KPI.
  • Well list and portfolio summary use current-state risk only. The detailed well page is richer because it also has cleaned history for stagnation analysis.
  • If backend 2 is down, the forecast source correctly drops to heuristic fallback. The lineage now makes that visible.
  • The commissioning source issue is still in the client data. The UI is only preventing bad display logic on top of that anomaly.
